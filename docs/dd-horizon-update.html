§1 — Goal & the never-delete invariant

We need a painless, repeatable way to roll the DD knowledge graph's version horizon forward to the latest released Data Dictionary, picking up new/changed paths, without reprocessing the whole graph and without ever losing a StandardName provenance link. The update must be incremental — LLM enrichment/refinement/embedding already done for unchanged nodes must persist via content hashes — while accepting that clustering is a full recompute each time.

The load-bearing guarantee is a single invariant:

THE INVARIANT — IMASNodes are never deleted by an update. An IMASNode's identity is its version-agnostic DD path (id = bare path, e.g. equilibrium/time_slice/profiles_1d/psi), written only via MERGE and never rewritten. A path removed in a newer DD version is marked (DEPRECATED_IN + lifecycle_status='removed'), never deleted. Because StandardName provenance links target IMASNodes by path, they survive removal for free — the old node persists, the link still resolves.

This is not aspirational — it holds in the live graph today (§2). The plan's job is to make rolling the horizon forward safe by construction and by test, and to forbid the two operations that would break it.

§2 — Current state (audit, 2026-07-22)

The working graph (codex) was loaded from a DD 4.0.1 export and SN work was layered on since. Concrete state:

  • Horizon: 35 DDVersion nodes present, 3.22.0 → 4.1.1 (DDVersion.id holds the version string). Config target [tool.imas-codex.data-dictionary].version = "4.1.1"; deps imas-data-dictionary>=4.1.1.
  • 4.1.0 is the last fully-built version — it alone carries build_hash, clusters_count=2891, clusters_built_at, and embeddings. The SN work is pinned to 4.1.0; its links are intact.
  • 4.1.1 was extracted but never finished — no build_hash, no clusters, no enrichment/embeddings for its new nodes. This is the horizon step to complete.
  • 61,366 IMASNodes; 9,480 embedded; 39,241 with null lifecycle_status (largely historical 3.x–4.0 nodes retained for lineage but never given full current-version treatment).
  • SN provenance edges to protect: 8,763 FROM_DD_PATH, 2,448 HAS_STANDARD_NAME, 1,154 source→VocabGap (HAS_STANDARD_NAME_VOCAB_GAP).
  • Invariant already holds: SN links resolve to removed IMASNodes (1,534), alpha (193), obsolescent (11) — removal does not break links. 0 dangling FROM_DD_PATH links today.
  • Reattach population: 2,696 RENAMED_TO edges; 496 SN sources are pinned to a path that was later renamed — the set the reattach policy (§5, D-reattach) must decide on.

The 13 red -m graph quality tests (embedding coverage, lifecycle, build_hash, clusters, unit integrity) are all symptoms of the unfinished 4.1.1 step + the stale-archive load — not the schema-compliance debt already fixed (commit ce452948).

§3 — How the mechanism already works (verified)

A code audit of the build pipeline (build_dd.py, dd_workers.py, dd_lifecycle.py, dd_graph_ops.py, cli/imas_dd.py) confirms the tooling already supports a safe incremental roll-forward. The figure shows the model; the findings below are the contract this plan relies on.

  1. Incremental is triggered by the build hash. _compute_build_hash hashes the schema version + sorted version list + ids filter. The up-to-date checks (_check_graph_up_to_date, _check_build_nodes_exist) compare it AND the version set; adding a version to the horizon changes both → extract+build re-run. No reset needed to roll forward.
  2. Removed paths are marked, never deleted. Per-version diffs MERGE a DEPRECATED_IN edge for removed paths; the end-of-build lifecycle reconciler (dd_lifecycle.py) stamps lifecycle_status='removed' for paths absent from the current DD, records renamed_to, and restores returnees — it never deletes.
  3. LLM work persists via three per-node hashes (enrichment / refinement / embedding). On a hash match the node advances its status without an LLM call; only a changed content string, changed model, or an explicit --reset-to re-runs the LLM.
  4. lifecycle_status + build_hash are repaired by the structural build, not gated by the LLM hashes. When phase_build runs (which it does whenever build_hash is null/stale or the version list changed), an unconditional final-pass SET re-stamps lifecycle_status for every current-version node and _write_build_metadata writes build_hash — while existing enriched/classified nodes keep their status (not dragged back through the LLM pipeline). So the stale-archive nulls get fixed for free on the next real build.
  5. Embedding is missing-only (claims refined embeddable nodes, re-embeds only where the embedding hash differs); clustering is a full recompute with label preservation (MERGEs cluster nodes keeping human labels, refreshes IN_CLUSTER, deletes only stale clusters) — and never touches an IMASNode.
  6. The only IMASNode deletes are _reset_to_extracted (guarded by --reset-to extracted / --force) and clear_dd_graph (the imas dd clear command). These DETACH DELETE and orphan every SN edge — forbidden by this plan's runbook.

Horizon caveat (design-critical): "current" is imas_codex.dd_version (the pyproject config), not max(versions). is_current, build-metadata stamping, and the lifecycle reconciler all key on it. To actually move the horizon forward you must bump the config to the target version; otherwise a newer version processed via --from-version is built as a DDVersion node but never becomes is_current, and paths valid only in that newer version get wrongly stamped removed by the reconciler.

§4 — The safe roll-forward runbook

Deliver a documented, repeatable procedure (and — D-ergonomics — optionally a thin imas dd update wrapper) that performs exactly this:

  1. Snapshot first (mandatory). imas-codex graph export — the belt-and-suspenders restore path (graph load) if anything goes wrong. A fresh snapshot imas-codex-graph-dev-ce45294.tar.gz already exists from 2026-07-22.
  2. Bump the horizon. Set [tool.imas-codex.data-dictionary].version to the latest installed DD release (verify against the installed imas-data-dictionary package; the resolver refuses a version beyond it). This re-keys is_current + the reconciler.
  3. Incremental build — never destructive. uv run imas-codex imas dd build --enrich-workers 6 (optionally --from-version <last-built> to bound the diff). Never --reset-to extracted, never --force, never imas dd clear. phase_build re-runs (build_hash stale) → repairs lifecycle + build_hash, MERGEs new paths, marks removed, hash- persists LLM work, embeds missing, and does a full re-cluster.
  4. Verify the invariant held (§5) before trusting the result.

Because enrichment/refinement persist via hashes and only the new/changed 4.1.x nodes need LLM work, the incremental cost is a fraction of a full rebuild; embeddings are free. Clustering re-runs in full (accepted).

§5 — SN-link safety, reattach, and the guardrail test

The invariant is enforced by construction (§3) but must also be enforced by test so a future refactor cannot silently break it:

  • Guardrail test (new, -m graph): capture the three SN-edge counts (FROM_DD_PATH, HAS_STANDARD_NAME, source→VocabGap) and the dangling-link count (SN source with dd_path set but no FROM_DD_PATH → IMASNode) before and after an update; assert edge counts are non-decreasing and dangling stays 0. This is the acceptance gate for any horizon update.
  • Reattach (D-reattach): 496 SN sources are pinned to a path later RENAMED_TO a successor. Default policy (matches the stated intent "the link should still point to the old version"): the old node persists, so links stay pinned — no forced follow. The reattach mechanism is a repair, not a migration: if an update ever leaves a dangling link (should be impossible under the invariant), a reconcile step re-resolves source.dd_path → IMASNode and, where the path was renamed, offers to re-point to the successor. Specify it, gate it behind an explicit flag, and prove it is a no-op on a healthy graph.
  • Forbidden operations (encode as a runbook guard / lint): --reset-to extracted, --force, imas dd clear — any of these on a graph carrying SN work is a hard stop; recovery is graph load from the snapshot.

§6 — Validation & the graph-quality gates

Acceptance = the guardrail test green PLUS the previously-red -m graph quality tests moving to green as a consequence of the finished horizon:

  • test_dd_build.py: embedding completeness, field lifecycle status — fixed once 4.1.1 embeds + the reconciler stamps lifecycle.
  • test_structural.py: current DD version has build_hash; clusters have members — fixed once build_hash is written for the new current version and clustering re-runs.
  • test_data_quality.py: StandardName embedding coverage, empty descriptions, COCOS half-state.
  • test_sn_unit_integrity.py: the ~248 SN-vs-DD unit mismatches are a separate curation axis (mostly V.m^-1↔m^-1.V ordering that normalizes to the same unit; some genuine, partly DD-side). Scope: a normalization pass for the ordering class; track the genuine mismatches separately — do NOT block the horizon update on them.

Record before/after counts for each gate and the incremental LLM spend vs a full-rebuild baseline (the headline "incremental, not full-reprocess" proof).

§7 — Ergonomics: the dd update affordance

Today the update is imas dd build (+ the config bump). To make it painless and hard to misuse, D-ergonomics decides between:

  • A documented runbook only (this plan's §4) — zero new code, relies on operator discipline to avoid the destructive flags.
  • A thin imas dd update command that: reads the latest installed DD version, bumps the config (or accepts --to-version), refuses if it would lower the horizon, runs the incremental build with the destructive flags structurally unavailable, and runs the guardrail check automatically. Safer by construction; the recommended deliverable.

§ Decisions

D-ergonomics — ship a dedicated imas dd update command (destructive flags structurally unavailable + auto guardrail check), or document the imas dd build runbook only?

D-reattach — for the 496 SN sources pinned to a renamed path: keep them pinned to the persistent old node (links stay stable), or follow RENAMED_TO to the successor node?

D-historical — enrich/embed/lifecycle-stamp only the current-version horizon (existing design; historical-only nodes keep null lifecycle/no embedding), or backfill lifecycle + embeddings across all retained historical versions?
