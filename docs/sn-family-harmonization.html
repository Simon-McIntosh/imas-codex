§0 — Problem & evidence

An expert reviewer flagged that the standard-names (SN) pipeline generates sibling (grouped) names independently, so a family of related quantities drifts apart in its documentation (and, separately, in structure — that part is owned by sn-structural-repair). The motivating family is mode_number: its poloidal and toroidal members share a parallel template, but the radial member diverges entirely.

namedoc openingshape
poloidal_mode_number"Poloidal mode number m is the dimensionless integer Fourier harmonic label…""{Axis} mode number {sym} is the dimensionless integer…"
toroidal_mode_number"Toroidal mode number n is the dimensionless integer Fourier harmonic that l…"same template ✓
radial_mode_number"Dimensionless non-negative integer labeling the radial branch of a linear M…"diverges ✗

The same opening-phrase drift recurs across essentially every multi-member family sampled (velocity, magnetic_field, diffusion_coefficient, the density_at_<locus> families): members describe the same quantity-shape but open with "{Axis} component of…", "Projection of the … vector onto…", "{Axis} {quantity} is…", "Signed scalar projection of…". The physics is usually right; the structure is inconsistent.

Root cause

  • Docs generation is strictly per-name (workers.py:6974-7108). It injects only domain-wide "nearby names" — never the actual sibling set — and the docs prompt (generate_docs_user.md) carries no parallel-structure directive. Each doc is a fresh LLM call blind to how its siblings read.
  • Name compose batches globally by (grouping_cluster × unit × species) (enrichment.py:447-482) — siblings only see one another if they happen to co-batch.
  • No operator inspects a family as a whole. Consolidation is per-name dedup; review is batch-aware for context but scores each name in isolation; refine pools are per-name. The only family-aware code, the children→parent grounding in _enrich_for_docs_gen (workers.py:6825-6874), is per-name, not group-aware.

§0c — Hard sequencing

sn-coverage-and-hygiene §1 (derived-parent enrichment) is a hard prerequisite: it writes real descriptions onto the placeholder parents that become the harmonization anchors, and populates source_domains (the legitimacy oracle for cross-domain families). sn-structural-repair should land before harmonization too, so families/anchors are clean (no member resolves to a superseded node via a stale edge). Order: §1 enrichment → structural-repair → harmonization.

§1 — Drift detection (deterministic, no LLM)

Build the worklist with a cheap per-family signal so only drifted families cost money. Validated live by research.

  • Docs opening-template signature — normalize description to the first N word-tokens (lowercased, digits→#); a family's doc_drift = 1 − max_cohort_size / member_count. 
    def doc_sig(desc, n=6):
    toks = re.findall(r"[a-zA-Z]+|\d+", (desc or "").lower())[:n]
    return " ".join('#' if t.isdigit() else t for t in toks)
# group family members by doc_sig; drift = 1 - largest_cohort / n_members
  • Name-pattern signature — the family's populated grammar-segment tuple + leading-token pattern; an extra non-grammar token (the mhd case) or a physical_base=NULL outlier is flagged for the structural-repair plan, not harmonized here.
  • Grouping key = the shared HAS_PARENT parent restricted to operator_kind ∈ {projection, qualifier, coordinate} — research-validated as far more reliable than physical_base (which is too coarse: legitimately different quantities share a base).

§1 deliverable — a sn harmonize --report (read-only) that ranks families by drift and emits the worklist. Done-when: the report reproduces the 164-set worklist; the mode_number family flags radial at 0.67; density_at_<locus> families flag at 0.89–0.90. This piece is not gated on §1 enrichment — it ships now.

§2 — Sibling-set assembly

Enumerate harmonizable families from the graph (live members only):

MATCH (c:StandardName)-[r:HAS_PARENT]->(p:StandardName)
WHERE r.operator_kind IN ['projection','qualifier','coordinate']
  AND coalesce(c.name_stage,'') <> 'superseded' AND c.description IS NOT NULL
WITH p.id AS parent, r.operator_kind AS opk,
     collect({id:c.id, desc:c.description, dom:c.physics_domain}) AS kids
WHERE size(kids) >= 2 RETURN parent, opk, kids

Reuse summarise_family() (search.py:1070, returns segment_distinct / physics_domain_distinct / lineage) and find_related(rel='grammar') (search.py:741) for the anchor's neighborhood. Parentless bare-base families fall back to physical_base grouping. Done-when: given a seed name, the operator returns its full live sibling set with the chosen grouping key.

§3 — Parent-anchored docs harmonization (core)

  • Anchor selection, in priority order: (1) the enriched derived/structural parent with real docs_stage='accepted' docs (the reason §1 must run first); (2) the member with highest reviewer_score_docs; (3) the longest non-placeholder description. No accepted member yet → see the open decision.
  • One group-aware LLM call, cache-friendly static-first ordering: static system = scoring rubric + an explicit "parallel structure: same opening noun-phrase template across siblings; vary ONLY per-axis/species physics and symbol; preserve each member's unit/COCOS/domain"; dynamic user = anchor doc + ALL members' current description+documentation + shared DD context. Extend _enrich_for_docs_gen (workers.py:6687) from per-name to whole-set grounding. Output schema: {revised:[{name,description,documentation}], unchanged:[name]}.
  • Write via persist_refined_docs() (graph_ops.py:10058) per member — snapshots prior docs to a DocsRevision {id:"{sn}#rev-{n}"} via DOCS_REVISION_OF, advances docs_chain_length, and clears reviewer_*_docs + embed_text_hash so REVIEW_DOCS re-scores and the embed worker re-embeds. No new review machinery — harmonized docs re-enter the gate exactly like a refine_docs output.

Done-when: a flagged family's members share a parallel opening structure and all pass REVIEW_DOCS; a DocsRevision snapshot exists per change.

§4 — Name fixes (limited, via the rename cascade)

For genuine semantic naming divergence within a family, drive the existing rename_cascade() (cascade.py:302) with --dry-run and the --include-accepted / --override-edits guards. Research-confirmed limits: projection & coordinate children do not cascade (cascade.py:138,:501); subject-stack names don't round-trip. So the malformed-parent, projection-child, and non-canonical renames are not in scope here — they belong to sn-structural-repair. This section is therefore a thin name-consistency audit that routes mechanical/structural renames to that plan and only applies the cascade for true semantic-naming choices.

§5 — Idempotent done-state

Mirror the existing claimed_at / SNRun audit patterns. Store on each member harmonized_at + harmonized_group_signature (a hash of sorted(member_ids) plus the sorted member-doc content hashes). Re-harmonize a family only when its signature changes (a member added/removed, or any member's docs edited). Done-when: the worklist query returns ∅ — every sibling-set is either below the drift threshold OR harmonized_group_signature == current_signature. A re-run over an unchanged family is a no-op.

§6 — CLI + (later) in-loop pool

sn harmonize mirrors the sn review / sn coverage command shape: scope flags (--ids / --domain / --physical-base), --dry-run, -c/--cost-limit, --report. Reuse the pool_adapter explicit-set seeding pattern (pool_adapter.py:22/103). A low-weight in-loop pool (so new families self-heal) is a follow-up once the CLI pass is proven.

§ Decisions

How to store the per-family harmonization done-state?

Research recommends the per-SN scalar pair — sufficient for the membership+content signature and avoids a new node type + LinkML churn. A SiblingGroup node is only worth it if we later want per-family drift history/telemetry.

When a family has no docs-accepted member yet (no anchor), what does the operator do?

Defer is the conservative default — it keeps harmonization downstream of the normal accept gate and avoids anchoring on un-reviewed prose. Dominant-cohort lets early families self-organize but risks anchoring on a poor template. Post-§1, most families will have an accepted parent anchor, so defer rarely blocks.
