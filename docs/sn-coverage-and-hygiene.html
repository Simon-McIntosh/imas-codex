§1 — Derived-parent enrichment (coverage)

Problem. 723 derived parents carry the placeholder description "(deterministic parent — description pending LLM enrichment)" and are stuck in a deadlock: a placeholder description excludes them from REVIEW_NAME (the claim requires a real, non-placeholder description), and with no review score they are excluded from generate_docs (gated on reviewer_score_name IS NOT NULL). They can neither be reviewed nor documented. 612 of the 723 have domain-bearing children (so they can also inherit a physics domain); 61 have children without domains; 50 have no children. This is the gating item for ~700 names' worth of catalog coverage — including area_at_plasma_boundary, the dog-food subject of sn-expert-hints.

Design. Add a derived-parent enrichment step that replaces the placeholder with a real description synthesized from the parent's children (their accepted descriptions + shared DD context + the parent's grammar), then routes the parent into the normal flow:

  • Why children, not the DD path: a derived parent is an abstraction over its composed children; the children carry the concrete physics and (for 612/723) the domain. The parent description is best synthesized as the common physical meaning across its children — exactly the grounding the enrichment prompt should receive.
  • Cannot reuse generate_docs: it is gated on reviewer_score_name, which needs a real description first — the deadlock. The enrichment must run before review, as its own step (a small LLM call grounded on children), writing description + embedding.
  • Then route to review: with a real description + embedding, the parent satisfies the claim_review_name_batch gate (already drops the placeholder); the review-aware promotion (from the Phase-4 work) sends it to drafted → review → accepted → docs-eligible.
  • Domain inheritance: fold in the existing primary-domain inheritance so the enriched parent is scoped (612 inherit from children; the 50 with no children are legitimately unscoped and excluded).

§1 deliverables:

  • §1a — an enrich_derived_parents step: claim placeholder-description derived parents (concurrency-safe), build a children-grounded enrichment prompt, write real description + embedding, inherit primary domain. Done-when: placeholder count trends to 0 for parents with children; the 50 childless ones are skipped, not enriched.
  • §1b — wire it into the loop (a pre-review pass) or as an on-demand CLI (sn enrich-parents); cost-capped. Done-when: a run enriches a batch and they appear at name_stage='drafted' ready for review.
  • §1c — verify end-to-end: enriched parents flow review → accept → docs and export to the catalog. Done-when: a sample of formerly-placeholder parents reach docs_stage='accepted' with sound, children-consistent docs.

§2 — Bare-bracket normalize at source (hygiene)

Problem (f-snph-025). The per-rotation _normalize_bare_doc_links pass (inside resolve_doc_links) runs once near the end of run_sn_pools, but docs written by in-flight generate/refine tasks after that pass escape it. The result is a small rolling residue of bare [name] brackets (≈2–50 per $50 cycle, scaling with cycle size) that is only swept by a later settled invocation — and occasionally a doc reaches accepted with a bare bracket (observed: plasma_resistance; the docs carrying docs_updated_at=null are the tell). Throughout the $500 docs campaign this was mitigated by running resolve_doc_links manually each cycle, holding the published graph at 0 — but the source timing gap remains.

Design. Close the gap so no doc can reach accepted with a bare bracket, eliminating the need for the manual per-cycle sweep:

  • §2a — normalize on the accept path: invoke _normalize_bare_doc_links (scoped to the node) in the docs accept transition (persist_reviewed_docs when it promotes to accepted), so a doc is normalized at the moment it finalizes regardless of when it was written. Done-when: a doc written late in a rotation and accepted in the same rotation has 0 bare brackets without any post-pass.
  • §2b — guaranteed final reconcile: ensure resolve_doc_links runs as a true final step after all pool tasks fully drain in run_sn_pools (not mid-drain), as a belt-and-suspenders net for the in-flight window. Done-when: a simulated late-write rotation ends with 0 bare brackets and no manual intervention.
  • §2c — regression test: a test that accepts a doc whose text contains a bare [name] and asserts it is normalized to a link or stripped before/at acceptance. Done-when: test green; the per-cycle manual resolve_doc_links in the operating loop becomes unnecessary.

Recommended order: §1 → sn-structural-repair → sn-family-harmonization.

§ Decisions

Run derived-parent enrichment as an in-loop pre-review pass, or as an on-demand CLI step?

In-loop keeps the backlog from re-accumulating as new derived parents are created; on-demand gives tighter cost control for a one-off drain of the 723. Possibly both: in-loop with a cost cap.

Primary fix site for the bare-bracket residue?

Accept-path is the tightest guarantee (no accepted doc can carry a bare bracket); the final reconcile is a cheaper belt-and-suspenders. Doing both is ideal; accept-path is the must-have.
