§1 — Problem and current baseline

Plan 23 remains a low-implementation, high-value backlog item: the survey pegs it at 0.12 because only one narrow fix is actually resolved today. Everything else is still a roadmap for bringing generated StandardNames up to catalog quality in content, links, and validation discipline.

  • Resolved already: the old GAP-3 unit-persistence concern is closed. Units now persist as unit plus HAS_UNIT, so Phase 1A no longer requires code changes.
  • Still open: the benchmark harness, targeted-path CLI, publish/tag cleanup, DD context amplification, COCOS-aware docs guidance, full documentation template rewrite, asynchronous link resolution, and A/B validation loop.
  • Critical framing from the markdown source: quality parity is not a prompt-only problem. It depends on measurable benchmarking, better graph context, richer examples, and a link-resolution lifecycle that the catalog can trust.

§2 — Open implementation work in Phases 0–2

  • Phase 0 — benchmark harness. Define a holdout set of DD paths with catalog counterparts, score generated output on length/equations/sign conventions/typical values/measurement sections/link hygiene/tags/validation/reviewer score, and keep the results in a repeatable script or tracked evaluation workflow. This is the required baseline before any other phase can claim improvement.
  • Phase 1B — targeted --paths mode. Add a targeted-path route to the SN generator so specific DD paths can be processed without going through the full extract query, classifier, or already-named filters.
  • Phase 1C — publish/tag migration. Update publish.py to group by physics_domain before stripping primary tags, then run the graph migration that prefixes bare links with name: and removes primary tags from existing names.
  • Phase 2A — rich DD context. Extend the DD extraction query so compose gets fuller path documentation, COCOS behaviour, coordinates, related paths, and structure context via the library layer rather than a chat-formatted MCP surface.
  • Phase 2B — COCOS injection. When a path is COCOS-dependent, the generated documentation must carry an explicit sign-convention paragraph rather than relying on the reviewer to infer it later.

§3 — Open implementation work in Phases 3–5

  • Phase 3A — documentation template rewrite. Replace the short 200–500 character target with required content sections: definition, governing equation, variable definitions, significance, measurement/calculation, typical values, sign convention when needed, and cross-references. The source explicitly treats content gates as more important than a hard minimum character count.
  • Phase 3B — full curated examples. Promote a small showcase set of complete catalog-quality examples while keeping a larger abbreviated vocabulary set for coverage. These examples must not overlap the holdout paths from Phase 0.
  • Phase 4A–4C — async link resolution. Add schema fields for link_status, link_checked_at, unresolved_links, and link_retry_count; let compose emit plausible name: links beyond the known-list constraint; and add a separate sn resolve-links worker with retry/failure semantics and publish-time filtering of unresolved entries.
  • Phase 5 — A/B validation. Re-run the holdout set after each major phase, then run a bulk generation sample (for example equilibrium) to compare the final quality profile against catalog baselines.

§4 — Constraints, dependencies, and documentation fallout

  • Library-first context. The source explicitly rejects using the MCP server formatting layer for DD context enrichment; the implementation should call the shared library/query layer directly.
  • Link resolution needs a terminal failure state. The plan is explicit that unresolved links cannot silently loop forever; failed is part of the intended lifecycle.
  • Publish/export safety matters as much as generation quality. Unresolved links must not appear in published output.
  • This plan still feeds later benchmark work. Plan 34 and the state-of-the-nation research already depend on Phase 0’s benchmark-harness concept, so the migration should preserve that relationship rather than bury it.
  • Documentation updates remain open. AGENTS guidance, the research quality-gap analysis, schema references, and publish.py docstrings all need to move when this work lands.
