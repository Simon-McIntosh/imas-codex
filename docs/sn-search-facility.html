§1 — What is already in place

Plan 40 is materially further along than plan 23 or plan 36, but the survey still marks it as only 0.55 implemented because the final call-site migration phase remains open. The HTML migration therefore preserves the real state: most of the design and public-tool surface are settled, but the pipeline has not fully moved to the new backing functions yet.

  • Phase 1 writer direction is clear. The plan fixes grammar decomposition by always populating per-segment columns, conditionally writing typed edges, and recording parser fallback explicitly.
  • Phase 2 search design is fully specified. Hybrid search, tiered grammar weighting, RRF fusion, fetch scoping, related-name buckets, name checking, and family summaries are all laid out against real graph evidence.
  • Phase 3 MCP palette parity is defined. The seven-tool StandardName surface mirrors the DD family, and the include_standard_names=True default is already treated as a verified precondition rather than future work.

§2 — The work that is still open

  • Phase 4 call-site migration is still open. The real remaining job is to replace legacy imports and helper names across standard_names/workers.py, standard_names/review/audits.py, standard_names/review/enrichment.py, and the nearby-search helper path in enrich_workers.py.
  • The alias bridge is temporary, not the end state. The plan explicitly allows one release of deprecated aliases, but they should only delegate to the new backing functions and then disappear after the migration window closes.
  • DD-only suppression is only solved at construction time. The plan documents that a background flip to dd_only=True can emit only a warning because FastMCP cannot unregister tools after startup. That limitation still exists until a larger follow-up lands.
  • Documentation and public naming cleanup remain part of the tail. AGENTS guidance, README tool tables, and the final removal of public _sn/_sns suffixes all stay tied to the last migration pass.

§3 — Key technical content preserved from the markdown plan

  • Empirical baseline: the source plan keeps the graph census, the writer root cause, the DD-tool parity table, and the naming inventory. Those details remain relevant because the HTML version should still explain why the search redesign exists.
  • Tiered grammar policy: Tier 1 segments (physical_base, subject, geometric_base) dominate; Tier 2 needs a Tier-1 anchor plus vector/keyword co-occurrence; Tier 3 never carries a result alone. This is the core design guard against x_component_of_* floods.
  • Seven-tool parity target: search_standard_names, fetch_standard_names, find_related_standard_names, list_standard_names, list_grammar_vocabulary, check_standard_names, and get_standard_name_summary remain the intended public family.
  • Open acceptance work: A11–A14 are the practical checks that still matter: flood resistance, bucketed related-name output, AST-based naming enforcement, and the DD-only construction-time gate.

§4 — Open decisions and follow-on choices

The source still leaves a small set of operational choices open even after the v3.2 review round.

  • Should true auto-detection for SN registration be added now, or deferred until FastMCP can safely unregister tools after startup?
  • Should the seeded tier weights and k_rrf=60 stay fixed until real traffic exists, or be tuned from a held-out query set immediately?
  • Should group_by_base=True under mode="vector" raise an error or quietly no-op?
  • When both return_fields and the fetch convenience flags are supplied, should the explicit whitelist win?
