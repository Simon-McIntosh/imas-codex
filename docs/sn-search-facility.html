SN Search Facility

Sprint S2 closed the remaining search-facility tail. The live code now matches the intended public surface: the seven StandardName MCP tools are backed by the canonical search and fetch helpers, the last in-scope callers were migrated, and the temporary alias bridge is gone.

§1 — Landed outcomes

  • Canonical backing is now the single source of truth. Search and fetch both flow through imas_codex.standard_names.search, including the MCP fetch wrapper.
  • Search semantics were tightened on the live surface. physics_domain is pushed into the canonical query paths, empty-query filter searches use the same backing logic, and result enrichment now carries the domain metadata the callers expect.
  • Fetch projection behavior is explicit and tested. return_fields remains the final whitelist over convenience flags, neighbour-only projections still expand neighbours, and aggregate-only projections keep one row per requested name.
  • The migration tail is gone. The remaining in-scope callers moved to the canonical helpers and the deprecated aliases were removed instead of being carried into another release.

§2 — Evidence recorded from sprint S2

Two close-out commits landed on main: 0b8e423b closed the canonical-backing, caller-migration, and fetch-wrapper gaps, and 810b1e0a restored the neighbour-projection and aggregate-only fetch semantics surfaced by code review.

The targeted search and MCP regression suite passed after the final fix: tests/standard_names/test_search_streams.py, test_sn_tools.py, test_search_facility_smoke.py, tests/llm/test_search_standard_names_segment_filter.py, test_sn_mcp_tools.py, test_tool_schemas.py, and test_public_api_naming.py. A combined search and fan-out regression pass also stayed green on main.

§3 — Deferred tail

  • DD-only auto-detect remains construction-time only. FastMCP still cannot unregister tools after startup, so background detection continues to warn rather than mutate the running tool palette.
  • Search tuning is deliberately conservative for now. The seeded tier weights and k_rrf=60 stay in place until a real held-out traffic/query set exists.

§ Decisions

Should plan 40 implement true SN tool auto-detection now?

The live runtime still cannot unregister tools after startup, so background detection remains advisory instead of mutating the running palette.

How should the initial tier weights and k_rrf be handled?

The facility is now correct on the live surface, but there is still no real held-out traffic set that would justify retuning the seeded defaults.

What should group_by_base do under mode="vector"?

That combination is not part of the current public API surface; if it returns, an invalid combination should fail loudly rather than silently changing behavior.

Which fetch argument should win when both whitelist and convenience flags are supplied?

The canonical fetch path now enforces return_fields as the final whitelist and carries the extra grouping data only internally when the projection needs it.
