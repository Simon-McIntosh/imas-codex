§1 — Problem & motivation

We have no robust way to apply expert pressure to a standard name or its documentation. When a reviewer (human) spots that a name is wrong or a doc is imprecise, the only options today are: (a) hand-edit the graph text directly — which decouples the edit from the pipeline's style rules, grammar validation, RD-quorum review, and scoring; or (b) revert to pending and hope the LLM independently arrives at the desired result. Direct edits are exactly what produced the rushed, off-house-style Alfvén rewrite earlier in this campaign; uncontrolled regeneration gives no steering at all.

The canonical motivating case surfaced this session is the plasma-boundary area family. The pipeline correctly produced a swept-surface quantity — flux_surface_area_at_plasma_boundary ("total geometric surface area of the last closed flux surface, computed by revolving the poloidal boundary contour through the toroidal angle") and surface_area_of_flux_surface. That leaves the bare area_at_plasma_boundary (currently a derived parent with a placeholder description) semantically ambiguous: "area" could mean the swept surface or the poloidal cross-section. The DD source distinguishes them — DD area is the poloidal cross-sectional area enclosed by the surface, DD surface is the swept toroidal surface. The right move is to steer the bare "area" toward an explicit poloidal cross-section framing (e.g. poloidal_cross_section_area_of_plasma_boundary) — a decision an expert should be able to propose and have the pipeline realize, score, and accept, rather than hand-patch.

Goal: a "hint" (proposal) structure that lets a human inject a suggested name or doc direction which is side-carried through the existing generate→review→score flow. The hint biases generation only; the pipeline's grammar checks, style rules, RD-quorum review, and numeric scoring remain the gate. A good hint is realized and accepted; a hint that violates the rules or is physically wrong is still caught and scored down. The proposal applies external pressure on direction without a full decouple from our checks.

§2 — Key insight: the rails already exist

The pipeline already threads a suggestion through regeneration — it just sources that suggestion from the reviewer LLM, not from a human. The RD-quorum reviewer emits an optional revised_name / suggested_name plus reviewer_comments; these are carried into the refine_name prompt (render_prompt("sn/refine_name_user", …), workers.py:5427) so the next compose attempt sees "the reviewer proposed X — improve toward it." Docs have the parallel path via refine_docs (workers.py:7623) using the docs reviewer feedback.

An expert hint is the same object from a different origin. Instead of inventing a new path, we generalize the existing suggestion-threading to accept a human-sourced proposal that rides the identical rails into the generate/refine prompt context, then flows through review and scoring exactly as a reviewer suggestion does. This keeps the blast radius small and guarantees the hint cannot bypass the checks — it enters where suggestions already enter, and is gated where results are already gated.

§3 — Design: the hint side-car

A hint is attached to a StandardName (or its StandardNameSource) and carried as an optional context field into the generation prompts. It never reaches the review or scoring prompts — review independence is preserved, so the reviewer judges the result blind to the steering that produced it.

Schema (LinkML → generated models)

  • name_hint — proposed name string or a free-text directional instruction ("frame as poloidal cross-section, not swept surface").
  • docs_hint — proposed doc direction ("state the swept-vs- cross-section distinction explicitly; cross-link the surface variant").
  • hint_reason — the expert's justification (carried into the prompt so the LLM understands the why, not just the what).
  • hint_origin — expert (vs the implicit reviewer origin of existing suggestions); recorded for provenance so we can audit which names had human steering.
  • hint_status — open → applied | rejected | exhausted, driven by the same accept/refine/exhaust outcome the result earns.

A dedicated Suggestion node + edge is the alternative to scalar fields; the scalar-field approach is preferred for v1 because it mirrors how reviewer_score_name/reviewer_comments_name already live as scalars on the node (one open decision below).

CLI / MCP surface

A single entry point attaches a hint and re-enters the name into the pipeline at the correct stage:

  • imas-codex sn hint <name> --name "<proposed>" --reason "<why>" — sets name_hint/hint_reason, reverts name_stage to drafted so review_name/refine_name re-process with the hint in context.
  • imas-codex sn hint <name> --docs "<direction>" --reason "<why>" — sets docs_hint, reverts docs_stage to pending so generate_docs re-runs with the hint.
  • An MCP write tool wraps the same operation so hints can be proposed from the assistant surface (the "as if the suggestion comes from us" requirement).

Worker prompt injection

The generate/refine prompt templates gain a single conditional block, placed after the rules/style/grammar sections so the hint is framed as a proposal subordinate to the rules, never an override:

  • generate_name_system / refine_name_* — render an {% if expert_hint %} block: "A domain expert proposes: <name_hint> (reason: <hint_reason>). Adopt this direction only insofar as it satisfies every grammar, style, and naming rule above; if the proposal conflicts with a rule, follow the rule and explain the conflict in your reasoning."
  • generate_docs_system / refine_docs_* — the parallel block for docs_hint.
  • Review prompts get nothing. review_names_* and review_docs_* are unchanged — the reviewer scores the result without knowing a hint existed, so the gate stays honest.

Flow & guardrails

The hinted name re-enters the pipeline and flows through the unchanged gates. The hint is consumed on acceptance and persists across refine rotations until the result is accepted or exhausted:

Guardrails that prevent decoupling:

  • The hint is a proposal in the prompt, explicitly subordinate to the grammar/style/naming rules; the compose grammar validator and validation_status gate still reject malformed names.
  • Review and scoring never see the hint, so a hint cannot inflate a score — a hinted result must earn acceptance on its own merits.
  • An exhausted hint (refine cap reached without acceptance) is surfaced, not silently dropped — signalling the proposal was either wrong or in tension with the rules, for the expert to revise.
  • Provenance: hint_origin=expert records which names carried human steering, distinct from pure-pipeline and derived origins.

§4 — Practicality verdict

Practical, and low-risk. Because the existing reviewer-suggestion threading already proves the rail (a suggestion shaping regeneration, gated by review), the refactor is additive: new optional schema fields, one CLI/MCP command, one conditional prompt block per generate/refine template, and the accept/exhaust status bookkeeping. No change to the review path, the scoring, or the claim/flow control. The hint cannot decouple from the checks by construction — it enters at the generation step and is gated by the same review the LLM's own output faces.

§ Decisions

Store hints as scalar fields on StandardName, or as a separate Suggestion node + edge?

Scalar fields mirror the existing reviewer_score_name/reviewer_comments_name scalars and keep v1 minimal; a Suggestion node would support multiple competing hints + richer history at more cost.

When both a human hint and a reviewer suggestion are present in a refine rotation, how do they combine in the prompt?

Expert intent is usually the stronger signal, but the reviewer feedback may catch a rule conflict the hint creates — both-shown is safer for honesty.
