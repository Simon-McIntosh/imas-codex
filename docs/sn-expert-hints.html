§1 — Problem & motivation

We have no robust way to apply human or agent pressure to a standard name or its documentation. Today the only options are: (a) hand-edit the graph text — which decouples the edit from the pipeline's style rules, grammar validation, RD-quorum review, and scoring (this produced the rushed, off-house-style Alfvén rewrite earlier in this campaign); or (b) revert to pending and hope the LLM independently lands the desired result (no steering). We want a third path: a tool — imas-codex sn edit — through which a human or an agent proposes a change that flows through the normal pipeline as a steered candidate and is gated by the same checks the LLM's own output faces.

Canonical motivating case (the dog-food demo, §6): the plasma-boundary area family. The pipeline correctly produced the swept-surface quantities (flux_surface_area_at_plasma_boundary, surface_area_of_flux_surface), leaving the bare area_at_plasma_boundary (a derived parent) semantically ambiguous. The DD distinguishes them: DD area = the poloidal cross-sectional area enclosed by the surface; DD surface = the swept toroidal surface. The bare "area" should be steered to an explicit poloidal-cross-section framing (e.g. poloidal_cross_section_area_of_plasma_boundary) — and sn edit should realize that through the pipeline, not by hand.

§2 — Key insight: the rails already exist

The pipeline already threads a suggestion through regeneration — it just sources it from the reviewer LLM, not from a human. The RD-quorum reviewer emits an optional revised_name / suggested_name + reviewer_comments, carried into refine_name (render_prompt("sn/refine_name_user", …), workers.py:5427); docs have the parallel path via refine_docs (workers.py:7623). An sn edit proposal is the same object from a human/agent origin. We generalize the existing threading rather than inventing a new path — small blast radius, and the proposal cannot bypass the checks because it enters where suggestions already enter and is gated where results are already gated.

§3 — Design

§3.1 — Two modes (flexible by intent)

sn edit supports two entry modes, because an editor sometimes knows the exact end state and sometimes only the direction:

  • Redesign (full proposal) → skip GENERATE, enter at REVIEW. The editor supplies a complete replacement name and/or documentation. The proposal IS the candidate: it is written as a drafted candidate and enters the review cycle directly (no LLM generation step — the human/agent is the generator). Review + score still gate it, so a malformed or low-quality redesign is still caught.
  • Hint (suggest a change) → injected into GENERATE, then review. The editor supplies a direction ("frame as poloidal cross-section, not swept surface"); it is injected into the generate/refine prompt, the LLM composes the specifics under the rules, and the candidate flows through review. For nudging when the exact string/text is better left to the composer.

Both modes are gated by the unchanged review + scoring. The difference is only where the candidate originates — from the editor (redesign) or from the LLM steered by the editor (hint).

§3.2 — The reason rides into REVIEW (breaks the pull-back)

Critical mechanic. There is an existing pull in the pipeline back toward a name's original/established variant: the grammar canonicalizer prefers established forms, and the reviewer — judging a changed candidate blind to intent — tends to score the deviation below the original it already validated, so refine rotations drag the name back. To counter this, the editor's --reason is passed to the review gate (both review_name and review_docs), not only to generate.

What review sees: the reason (the intent and justification), framed as "a domain expert has deliberately steered this name/doc for the following reason: <reason>. Judge the candidate on its physical and grammatical merits given this intent; do not penalize it merely for differing from a prior variant." What review does not get: a literal "accept this" instruction or the proposed string presented as pre-approved — the reviewer still scores quality independently and can reject. This is the minimal context needed to neutralize the revert-to-original bias while keeping the gate honest. (This refines the earlier "review stays fully blind" stance: review stays blind to the proposal as an answer, but is told the reason a change is intended.)

§3.3 — CLI surface

The SN to edit is the positional argument. Mode flags are mutually exclusive; --reason is mandatory for every edit.

imas-codex sn edit <standard_name> (--hint TEXT | --rename NAME | --docs TEXT) --reason TEXT
                       [--axis name|docs|both] [--scope self|family|subtree] [--dry-run]

  positional
    standard_name            The SN id to steer (required).

  mode (exactly one)
    --hint TEXT              Suggest-a-change: direction injected into GENERATE; candidate then reviewed.
    --rename NAME            Redesign the name: full replacement; SKIPS generate, enters at REVIEW_NAME.
    --docs TEXT              Redesign the docs: full replacement documentation; SKIPS generate, enters REVIEW_DOCS.
                            (--hint may carry name and/or docs direction via --axis; --rename/--docs are full redesigns)

  always required
    --reason TEXT           Justification. Rides into GENERATE (hint mode) AND the REVIEW gate
                            (both modes) to break the pull-back to the original variant.

  axis
    --axis name|docs|both   Which axis the edit applies to (default inferred from the mode flag).

  family propagation (see §3.5)
    --scope self|family|subtree
        self      Only this SN re-enters the pipeline.
        family    This SN + its siblings re-derive together + parent re-derives (shared-base edits).
        subtree   This SN (a parent) + all HAS_PARENT children re-derive from the new base.
        (default: subtree when the SN is a derived parent, self when it is a leaf —
         a sibling-desync guard blocks a self-scoped edit that would fragment a family
         and tells you to pass --scope family)

  other
    --dry-run               Print what would re-enter the pipeline, at which stage, and the cascade — no writes.

An MCP write tool mirrors this exactly so an agent can propose edits from the assistant surface ("as if the suggestion comes from us"). The CLI and MCP tool share one implementation.

§3.4 — Schema

  • edit_mode — hint | rename | docs (records how the candidate originated).
  • name_hint / docs_hint — the proposed direction (hint mode) or the full replacement (redesign mode).
  • edit_reason — the justification; injected into generate (hint) and review (both modes).
  • edit_origin — human | agent (vs the implicit reviewer origin of existing suggestions); provenance for which names carried external steering.
  • edit_status — open → applied | exhausted | rejected, driven by the result's accept/refine/exhaust outcome.

Open decision below: scalar fields (mirrors existing reviewer scalars) vs a dedicated Edit/Suggestion node.

§3.5 — Family handling: parents, children, siblings

Derived parents are abstractions over composed children (the children are the real names; the parent is re-derived from them each rotation by the admission gate). An edit must keep a family internally consistent:

  • Editing a parent (the plasma-area case): default --scope subtree. The new base (e.g. area → poloidal_cross_section_area) cascades to every HAS_PARENT child, which re-derives its name/docs from the new base (radial_area_at_plasma_boundary → radial_poloidal_cross_section_area_at_plasma_boundary) and re-enters the pipeline with the same reason. The parent re-proposes; children follow.
  • Editing a leaf child: default --scope self. But a change to a component shared with siblings (the base quantity, a qualifier the family carries) would fragment the family if applied to one child only. The tool detects shared-segment edits and the sibling-desync guard blocks a self-scoped edit in that case, directing the editor to --scope family, which re-proposes the whole sibling set together and re-derives the parent.
  • Child → parent propagation is partly automatic: because the admission gate re-derives parents from their children each rotation, once an edited child (and its siblings under --scope family) land, the parent re-derives consistently on the next rotation. The tool's job is to ensure the right set re-enters together so the re-derivation sees a coherent family, never a half-edited one.
  • Superseded chain: a name change supersedes the old SN (existing REFINED_FROM mechanism); inbound doc links to the old name are rewritten to the accepted successor by the existing resolve_doc_links pass — so the family stays linked.

§3.6 — Worker injection points

  • Hint mode: inject name_hint/docs_hint + edit_reason into the generate/refine prompt context (generate_name_system, refine_name_*, generate_docs_system, refine_docs_*) as a block AFTER the rules, framed as a proposal subordinate to them.
  • Redesign mode: write the proposed name/docs as a drafted candidate (name change supersedes via the refine persist path), set the stage so REVIEW claims it — no generate.
  • Both modes: inject edit_reason into the review prompts (review_names_*, review_docs_*) as the intent context per §3.2. This is the one review-prompt change in the whole design.

§6 — Dog-food demo: plasma-boundary area

Graph-reality update (2026-07-03, pre-implementation audit). The target drifted since this plan was written: the SN is now area_of_plasma_boundary (the _at_→_of_ locus migration), it is a leaf (no HAS_PARENT children — not a derived parent), and it sits at name_stage=superseded with no successor: review disagreement (mean 0.846, threshold miss) killed it on exactly the ambiguity this plan describes — the reviewer's own comment reads "ambiguous between cross-sectional area enclosed by the boundary and toroidal boundary surface area". The DD path equilibrium/time_slice/global_quantities/area consequently has no live standard name. The demo therefore also exercises editing a name in a terminal stage (resurrect-by-redesign). The originally proposed target poloidal_cross_section_area_of_plasma_boundary does not parse under the current ISN grammar; poloidal_cross_section_of_plasma_boundary does (projection=poloidal, base=cross_section, locus=of plasma_boundary), with cross_section_of_plasma_boundary as fallback. Scope: --scope self (leaf); the subtree cascade is validated by §5f tests against a derived parent with children instead.

The end-to-end acceptance test. Steer the ambiguous bare area to an explicit poloidal cross-section through the pipeline: 

imas-codex sn edit area_of_plasma_boundary \
    --rename poloidal_cross_section_of_plasma_boundary \
    --reason "DD 'area' is the poloidal cross-sectional area enclosed by the boundary, \
              NOT the swept toroidal surface (that is surface_area_of_flux_surface). \
              The bare 'area' is ambiguous; name it explicitly as the poloidal cross-section." \
    --scope self

Expected behavior, demonstrating each design element:

  • Redesign mode writes the proposed name as a candidate and enters REVIEW_NAME — no generate step.
  • Reason → review tells the reviewer the rename is a deliberate disambiguation, so it does not pull back to the bare area_of_plasma_boundary.
  • Terminal-stage resurrection: the superseded, successor-less SN gains a successor via REFINED_FROM, re-covering the orphaned DD path.
  • Gates still apply: grammar validation, RD-quorum review, scoring — the rename only lands if it passes them.
  • Control edit: a deliberately-bad edit (physically wrong reason/name) is also submitted; it must score below threshold, proving the anti-revert directive does not rubber-stamp.
  • Result: the bare-"area" ambiguity is resolved by a steered candidate that flowed through generate(skipped)→review→score, with provenance edit_origin recorded — proving the tool works.

Prerequisite note (resolved): sn-coverage-and-hygiene shipped 2026-06-25 (enrich_parents pool; 854 derived parents documented), so the derived-parent backlog no longer blocks this demo.

§ Decisions

Store edit fields as scalars on StandardName, or as a separate Edit/Suggestion node + edge?

Scalars mirror the existing reviewer_score_name/reviewer_comments_name pattern and keep v1 minimal; edit history via a node can come later if competing proposals become real. Deducible v1-minimal default per the plan's own rationale.

How strongly should the reason steer review — neutral intent context, or an explicit instruction not to favor the prior variant?

The plan's §3.2 prose already words the review injection as the anti-revert directive ('judge on merits given this intent; do not penalize merely for differing from a prior variant') — locking to match. Guard against rubber-stamping retained: §6 runs a deliberately-bad control edit to verify scores still discriminate. Graph evidence supports the need: area_of_plasma_boundary died in review (disagreement, superseded, no successor) precisely on the ambiguity the reviewer itself named.

Default scope for a shared-base leaf edit — block and require explicit --scope family, or auto-promote to family?

Blocking a shared-base leaf edit and requiring explicit --scope family avoids silently rewriting a sibling family; matches the plan's lean and the existing cascade.py safety-knob style (override_edits / include_accepted opt-ins).
