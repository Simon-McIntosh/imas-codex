Standard-Name Graph Evaluation

Read-only graph snapshot taken at authoring time during a live sn run campaign (codex graph, location iter). Counts are live head; treat them as a moving baseline. This is an analysis-only assessment — no code changes were made; all findings are reported for triage.

Headline verdict. Sample quality is high — 28/28 sampled accepted names round-trip cleanly through the ISN grammar (100% grammatical canonicity), and reviewer scores on the 54 machine-reviewed accepted names are strong (median 0.969, mean 0.957, min 0.85). The defects are semantic, not grammatical: a handful of accepted names drop a load-bearing qualifier (e.g. radiation/nuclear) and/or carry documentation that over-claims relative to the DD source path. Two confirmed duplicate pairs share one DD source.
Linkage verdict. StandardName-level HAS_ERROR = 0 is expected, by design: error/uncertainty quantities are intentionally not given their own catalog names or HAS_ERROR edges — they are handled universally by applying an error-modifier to the existing base name (e.g. uncertainty_of_<name>), deliberately avoiding the cost of generating/reviewing per-error names. HAS_ARGUMENT = 0 is separately expected — genuinely not-yet-exercised: no binary-operator (two-argument) names have been sourced, and the derivation has no HAS_ARGUMENT edge type (binary arguments map to HAS_PARENT). The HAS_PARENT orphans are by design — the derived-parent admission gate deliberately drops bare-base parents. The real linkage question, the HAS_PARENT orphans, resolves to expected behaviour with one watch-item (species-stripped intermediates not materialised; §3.2).

1.2 — Adversarial sample (28 names across domains)

Twenty-eight accepted names were sampled across the domain-sorted list and assessed for (a) grammatical canonicity via ISN parse→compose round-trip, and (b) description/documentation fidelity against the source DD path's own documentation.

  • Grammatical canonicity: 28/28 PASS. Every sampled name round-trips exactly through the ISN grammar (parse → compose returns the identical string). No decomposition defects, no unregistered tokens in accepted names.
  • Description fidelity: ~24/28 clean; 4 with semantic defects. The defects are concentrated in two patterns: (i) a load-bearing qualifier dropped from the name while the docs silently re-assert it, and (ii) docs that over-claim scope relative to a DD path that is explicitly narrower (e.g. a _thermal path described as covering thermal + fast).

1.3 — Worst 5 defects (with concrete reasoning)

1. temperature_at_magnetic_axis — mislabeled near-duplicate, docs contradict the DD source.
Source DD path ece/t_radiation_central (unit eV), DD documentation: "Radiation temperature from the closest channel to the magnetic axis." The accepted SN doc instead asserts "electron kinetic temperature… distinct from a blackbody-[radiation] temperature." An ECE channel measures radiation (brightness) temperature — equal to electron temperature only in the optically-thick limit. The name drops the radiation qualifier entirely, and the doc actively re-labels the quantity as something the DD says it is not. Worse: the sibling radiative_temperature_at_magnetic_axis exists in the same family and is sourced from the same DD path — so this is a duplicate of a correctly-named node. Consolidation should have caught it.
2. ion_power_density — duplicate DD source + scope over-claim.
Source DD path waves/coherent_wave/profiles_1d/ion/power_density_thermal, DD doc: "Flux surface averaged absorbed wave power density on the thermal species." The accepted SN doc claims "Total volumetric auxiliary-heating power source for all ions… encompassing thermal bulk and fast ion contributions." The DD path is explicitly _thermal (thermal species only) — the doc over-claims by adding fast-ion contributions the path excludes. The sibling ion_wave_absorbed_power_density is sourced from the same DD path: a second duplicate pair on one DD source.
3. power_density_at_midplane — generic name for a domain-specific quantity.
Source DD path breeding_blanket/module/layer/time_slice/midplane_power_density_nuclear, DD doc: "Nuclear heating power density in the layer, evaluated at midplane." The accepted name drops both nuclear and heating, leaving a generic power_density_at_midplane that any midplane power density could match. The refine chain already contains the correct successor nuclear_heating_power_density_at_breeding_blanket_midplane (drafted) — so this is a known-weak name caught mid-refinement, but it is still in accepted name_stage with docs_stage=drafted.
4. source_over_divertor_region_due_to_recycling — bare source physical-base, no source-path link.
This accepted name has no PRODUCED_NAME link to a StandardNameSource (it is a refine-chain product). source as a bare physical_base is semantically thin ("source of what?"); the meaning rides entirely on the _due_to_recycling mechanism. It is grammatically valid but reads as under-specified against its sibling neutral_particle_flux_over_divertor_region_due_to_recycling, which names the quantity (neutral_particle_flux) explicitly.
5. ion_power_density family — broad scope-attribution prose across the auxiliary-heating cluster.
Several accepted auxiliary-heating densities (ion_absorbed_power_density, ion_power_density_due_to_wave_particle_interaction, ion_species_density) carry docs that say "summing over both thermal bulk and suprathermal fast-ion" / "includes thermal, fast, and suprathermal." Individually defensible, but as a group the docs assert a thermal+fast union that is not always backed by the specific DD leaf the name is sourced from. This is a systematic docs-review blind spot worth a targeted re-review of the auxiliary-heating power-density family, not a per-name grammar problem.

Quality verdict: grammar pass rate 28/28 (100%); description-fidelity pass rate ≈ 24/28 (≈86%) on the sample, with the failures clustered in two recurring patterns (dropped distinguishing qualifier; thermal-vs-fast scope over-claim) and two duplicate pairs sharing a single DD source. None of the defects are grammatical; all are semantic and concentrated in the diagnostic-temperature and auxiliary-heating-power-density families.

2.3 — Prompt robustness (exhausted-name failure patterns)

The 20 exhausted names (failed to reach acceptance after max refine rotations) have a clear recurring signature in their reviewer comments. The dominant failure is not a prompt-quality problem — it is the compose/refine prompt being asked to name a concept whose required token is missing from a closed segment, and the LLM responding by absorbing the missing token into an illegal compound physical_base instead of emitting a VocabGap.

Robustness finding. The recurring pattern is that the compose prompt, when it cannot find a token, fabricates an illegal compound (e.g. fast_pressure, confinement_time, gap_distance) rather than emitting a VocabGap. The reviewer reliably catches it (low scores, explicit "should be vocab_gap" comments), so quality is protected — but the pipeline wastes refine rotations on un-nameable concepts. Recommendation: strengthen the compose-prompt rule to emit a VocabGap when the only way to use a concept is to invent a compound physical_base/subject, and add the high-frequency offenders (fast subject, surface/passing/co modifiers) to the closed-segment vocab via the ISN fork rotation if they are physically warranted. This converts wasted refine cost into actionable gap signal.

So the "HAS_ERROR=0" and "HAS_ARGUMENT=0" of the brief are about StandardName→StandardName edges. Both zeros are expected, for distinct reasons:

HAS_ERROR = 0 — EXPECTED, BY DESIGN (universal error-modifier strategy).

This is a deliberate design choice, not a defect. Error and uncertainty quantities are intentionally not given their own catalog names or per-name HAS_ERROR edges. Uncertainty is handled universally by applying an error-modifier to the existing base name (e.g. uncertainty_of_<name> / upper_uncertainty_of_<name>), so an error quantity is expressed as a modification of its parent rather than as a distinct first-class standard name. Generating and reviewing a separate standard name for every _error_upper/_error_lower/_error_index DD field would burn generation and review cycles for little catalog value — error names probably do not need to live in the catalog at all. The state is revisitable later, but zero SN-level HAS_ERROR edges is the intended current state.

Context (for completeness, not a defect): the DD graph does carry the raw material — 31,281 DD _error_ IMASNodes and 23,006 IMASNode→IMASNode HAS_ERROR edges; 137 of 171 composed/attached DD sources have ≥1 error child in the DD graph. That raw material is exactly what the universal error-modifier strategy avoids materialising as separate catalog names. The deterministic minting machinery (error_siblings.mint_error_siblings and its workers.py gate) is consistent with this: it is not currently producing SN error nodes, which matches the design intent. No fix attempted.

HAS_ARGUMENT = 0 — NOT-YET-EXERCISED (no bug, verified on its own merits).

There is no HAS_ARGUMENT edge type in derivation.py at all. The current derivation emits HAS_PARENT (binary operators would produce two HAS_PARENT edges, one per argument — see derivation._derive_structural), HAS_ERROR (uncertainty), and HAS_LOCUS (locus grouping). I checked whether any argument-bearing / parameterised names exist in the sourced set and whether the binary-operator branch would fire: ran derive_edges() over the 35 names matching binary/argument markers (ratio_of_, _to_, difference_of_, product_of_, _over_). None produced a binary operator edge. The _due_to_ mechanism and _over_/_at_ loci are all handled as qualifier/locus HAS_PARENT edges, not binary arguments. Notably the value-parameterised q95 name safety_factor_at_normalized_poloidal_magnetic_flux_equal_to_0_95 derives a locus HAS_PARENT → safety_factor (plus HAS_LOCUS), not an argument edge. So zero is expected: no input name uses a true two-argument binary operator, and the relationship — if one ever did — would be recorded as HAS_PARENT with operator_kind="binary", not a dedicated HAS_ARGUMENT edge. No action needed.

3.2 — HAS_PARENT orphans — root cause (by design)

37 of 99 accepted names have a HAS_PARENT edge; 62 do not. Among accepted names that carry a subject segment, 7 have no HAS_PARENT: ion_mass, ion_temperature, neutral_mass, and the four toroidal_deuterium_tritium_velocity_at_* names.

Investigation. Running derive_edges() directly shows the derivation does emit the edge: derive_edges("ion_temperature") → HAS_PARENT → temperature (qualifier peel). So the logic is not failing to fire. The reason the edge is absent from the graph is the derived-parent admission gate (parents.is_admissible_parent_name, applied in graph_ops._filter_admissible_parents): it deliberately drops HAS_PARENT edges whose target is a bare base with no specificity and no vector-like topology. Confirmed:

Candidate parentis_admissible_parent_name
temperatureadmit=False — "bare base, no qualifier/locus/projection/operator/mechanism"
massadmit=False — bare base
velocityadmit=False — bare base
elongationadmit=False — bare base
toroidal_velocityadmit=True — "has projection (axis=toroidal)"
electron_temperatureadmit=True — "has qualifiers [electron]"
plasma_currentadmit=True — "has qualifiers [plasma]"

The bare-base targets temperature/mass/velocity/elongation do not exist as nodes either — they were never created, precisely because the gate refuses to admit them. The control names electron_temperature, electron_density, ion_density are also parentless for the same reason (their would-be parent is the inadmissible bare base).

HAS_PARENT-orphan verdict: BY DESIGN, not a derivation bug. The admission gate (parents.py clause A/B) intentionally prevents generic root nodes like temperature from existing and grouping unrelated names. ion_temperature being parentless is the gate working as intended. The only watch-item: the four toroidal_deuterium_tritium_velocity_at_* names emit a HAS_LOCUS edge (to the locus, e.g. magnetic_axis) instead of HAS_PARENT, because the locus peel fires before the species/qualifier peel and the species-stripped intermediate (toroidal_velocity_at_magnetic_axis) is not in the graph. They are grouped by locus, not by quantity — defensible, but if quantity-grouping is desired the intermediate names would need to exist.

3.3 — REFINED_FROM (23) — coherent

All 23 REFINED_FROM chains are well-formed: edge direction is new→old, and every predecessor (old) is name_stage='superseded' (23/23). Chains are sensible refinement sequences, e.g. li → normalized_internal_plasma_inductance → normalized_of_internal_inductance_of_plasma; safety_factor → field_line_safety_factor_of_flux_surface → safety_factor_of_flux_surface (accepted tip); particle_flux_over_divertor_region_due_to_recycling → total_particle_flux_over_divertor_region_due_to_recycling. No reversed edges, no live predecessors, no self-loops.

3.4 — Family grouping — coherent, with the §1 duplicates as the exception

Three physical_base families inspected (pressure, temperature, density). HAS_PARENT chains within families are correct and group by quantity, not noise:

  • density: ion_state_energy_density → energy_density, poloidal_current_density → current_density, toroidal_current_density → current_density, total_ion_density → ion_density, thermal_neutral_density → neutral_density, ion_absorbed_power_density → absorbed_power_density. All admissible-parent links, all sensible.
  • pressure: coolant_pressure_at_cooling_circuit_inlet → pressure_at_cooling_circuit_inlet, perpendicular_fast_ion_state_pressure → fast_ion_state_pressure. Correct.
  • temperature: radiative_temperature_at_magnetic_axis → temperature_at_magnetic_axis. Correct structurally — but this is exactly where the §1.3 defect lives: temperature_at_magnetic_axis is itself a mislabeled duplicate of the same DD source.

Duplicate/synonym check: two confirmed duplicate pairs, both sharing one DD source — (temperature_at_magnetic_axis, radiative_temperature_at_magnetic_axis) on ece/t_radiation_central, and (ion_power_density, ion_wave_absorbed_power_density) on waves/.../ion/power_density_thermal. The consolidation pass should fold each pair to the better-specified name and supersede the other.
