SN Benchmark Evolution Strategy

This HTML migration keeps plan 33 in design / research state. The implementation fraction remains 0.05: the problem statement, phases, and acceptance criteria are explicit, but the code and benchmark integration work were still queued.

§1 — Why the plan exists

The benchmark had converged into a narrow score band and still suffered from single-reviewer bias, sparse and biased anchors, coarse grammar docking, and isolation from the production workflow. Plan 33 exists to make the benchmark discriminate better, mirror production review policy, and become operationally useful.

§3 — Current blockers

  • Reviewer bias persists. Single-model scoring still tilts the benchmark away from real disagreement patterns.
  • Anchor sparsity persists. Most physics domains remain unrepresented in calibration anchors.
  • Workflow integration is absent. The benchmark still lacks graph persistence, status visibility, grammar-sync hooks, and the planned regression gate.

§4 — Plan 34 dependency

Plan 33 directly informs and partially blocks the benchmark-v1 build-out, especially where the eval set wants multi-reviewer consensus, grammar-equivalence scoring, and expanded anchor coverage.
