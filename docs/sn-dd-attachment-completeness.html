§1 — The gap: an accepted name attaches only to where it was born

A StandardName is a concept — ion_atomic_number, electron_temperature, poloidal_magnetic_flux. The IMAS DD contains that same physical quantity at many paths across many IDSs (every z_n leaf is an atomic number; every Langmuir-probe t_e is an electron temperature). The (:IMASNode)-[:HAS_STANDARD_NAME]->(:StandardName) edge is what turns the standard-name vocabulary into a navigation index over the DD: given a name, list every DD path that realizes it; given a path, read its canonical name.

Today that index is almost empty. The generate pipeline writes a HAS_STANDARD_NAME edge only from the single source that seeded the name (plus whatever consolidation happened to touch), so the concept never reaches its siblings. The live graph:

Lineage — dependency satisfied. The units_agree gate this plan relies on has shipped: the unit-curation axis (followup f-ddhu-004 from dd-horizon-update) landed as 7d9648e7 (imas_codex/units/dd_unit_exceptions.py — units_agree(sn_unit, dd_unit, dd_path) over the ISN canonical_unit authority + a path-globbed DD-unit-bug exception record), with the HAS_UNIT self-heal (30ba5053) and the source_paths↔edge reconcile (b955da37). This plan is therefore unblocked. It was originally sketched as f-sdu-005 inside that campaign and is promoted here to its own plan.

§1b — Root cause: the DD-side edge was written for the birth source only

The generate write path (create_nodes, graph_ops.py:2884) MERGEs exactly one HAS_STANDARD_NAME edge per name record — to the single representative source_id. When consolidation collapses N same-concept DD sources into one StandardName, the other N−1 keep their PRODUCED_NAME/FROM_DD_PATH provenance but never receive a DD-side edge, and nothing downstream backfills them (refine migrates the one edge, prune deletes, the source reconcile only touches FROM_DD_PATH). The gap is provenance-derivable and exact: 

MATCH (s:StandardNameSource)-[:PRODUCED_NAME]->(sn:StandardName)
WHERE sn.name_stage='accepted'
MATCH (s)-[:FROM_DD_PATH]->(dd:IMASNode)
WHERE NOT (dd)-[:HAS_STANDARD_NAME]->(sn)
RETURN count(*)  -- 3423 edges · 1522 names · 3348 DD nodes

The tell is an asymmetry: the SN-side source_paths scalar is already reconciled from provenance (reconcile_standard_name_source_paths unions PRODUCED_NAME source ids, graph_ops.py:7203) — which is why it correctly reads 85. The DD-side HAS_STANDARD_NAME edge is the one projection that was left imperative and incomplete.

§1c — The published catalog is unaffected; the edge is an internal concern

The exported catalog carries the DD↔path mapping, and it is complete. The imas-standard-names-catalog site is a faithful render of the catalog, built in that repo's CI from ISN + the catalog YAML with no graph access. The chain that puts dd-paths on the page:

  1. Codex run_export populates each entry's sources list via _fetch_sources_for_entry → fetch_public_semantic_sources (export.py:1470, default on).
  2. That function walks PRODUCED_NAME (provenance_lifecycle.py:42) — not HAS_STANDARD_NAME — emitting per-source dd_path, dd_version, dd_documentation, and enrichment. Provenance is complete (85/85 for ion_atomic_number), so the exported sources are complete.
  3. ISN yaml_store strips only the raw dd_paths/physics_domain scalars, keeping sources; ISN catalog/dataset.py — the "SPA dataset builder" — reads sources to build the site data.

So the missing HAS_STANDARD_NAME edges do not touch the published catalog or its SPA. The edge is instead read pervasively inside the pipeline: review-context enrichment (which DD nodes back a name, fed to the reviewer — review/enrichment.py:46), source eligibility (already-mapped nodes are skipped — sources/dd.py:584), coverage, dedup, derived-parent admission (parents.py), plus the codex MCP edge-walkers (reverse DD-path → name, find_related, source_ids columns). At 15.5% edge coverage these all under-read for the 1522 affected names. That — not the catalog — is the blast radius.

This also settles "are the three edges redundant?": PRODUCED_NAME+FROM_DD_PATH is the complete authority and already drives the catalog; HAS_STANDARD_NAME is a denormalized 1-hop convenience. But it is load-bearing in dozens of internal queries, so dropping it (repointing every consumer to the 2-hop) is infeasible — the sound move is to keep it and derive it from the authority.

§2 — Design: one authority, self-healing projections

The root cause is that HAS_STANDARD_NAME was treated as authored truth, written imperatively one-edge-per-name. The fix is to treat it as what it structurally is — a materialized projection of the per-source provenance — and reconcile it, exactly as source_paths is already reconciled. This gives a single authority and two self-healing views, so the 3423-edge drift becomes structurally unrepresentable.

ElementRole
FROM_DD_PATH + PRODUCED_NAME Authority — per-source provenance; the pipeline writes it; already complete (85/85 for ion_atomic_number).
HAS_STANDARD_NAME Materialized view — reconciled each run from the 2-hop, units_agree-gated; the codex-internal 1-hop index for MCP / graph-RAG / coverage.
source_paths scalar Materialized view — SN-side, already reconciled from provenance (b955da37).

The two projections are not redundant with the authority — they are denormalized indexes it would be expensive to recompute per query. The authority is not redundant with them — it is the only per-source record. What was wrong was maintaining one projection by hand.

Two populations of missing edge, one gate:

  • Provenance backfill (the 3423 — deterministic, $0). For every (s)-[:PRODUCED_NAME]->(sn) where (s)-[:FROM_DD_PATH]->(dd), MERGE (dd)-[:HAS_STANDARD_NAME]->(sn) — gated on DD-eligibility (node_category ∈ SN_SOURCE_CATEGORIES) and units_agree. A pair whose units disagree is dropped and logged to the unit-curation triage, never forced: a disagreement means a wrong name or a wrong source, not an edge to write. No LLM, no review.
  • True discovery (measure first, maybe empty). A DD-eligible node that is the same quantity but has no producing source (never extracted, or extracted to a different name) is invisible to the provenance backfill. But most siblings are their own source (every z_n path is a StandardNameSource that composed to ion_atomic_number), so the "→ all z_n" case is largely served by the backfill already. Measure the residual before building any predicate. If a genuine gap exists, close it by extracting a source for that node — so provenance stays the single authority — never by authoring an orphan HAS_STANDARD_NAME edge with no source behind it.

§3 — Implementation approach

A single idempotent reconcile sub-step on every sn run post-drain (right beside reconcile_standard_name_source_paths in loop.py), plus an sn run --only attach entry point for a one-shot backfill of the current backlog. Idempotency is mandatory — a second run writes zero new edges. It mirrors the existing reconcile pattern in graph_ops.py; no one-off migration script (banned per AGENTS.md).

  1. Materialize the edge — one Cypher pass: MERGE (dd)-[:HAS_STANDARD_NAME]->(sn) for every provenance pair on a non-terminal name, skipping DD-ineligible nodes and dropping (with a logged reason) any pair failing units_agree.
  2. Reconcile the scalar — the existing reconcile_standard_name_source_paths then runs unchanged; edge and scalar agree by construction because both derive from the same authority.
  3. Guardrail (acceptance oracle) — extend the SN-link integrity test: after a run, HAS_STANDARD_NAME equals the eligible ∩ units_agree provenance projection, edges are non-decreasing, zero dangling, and every edge satisfies units_agree.
  4. Measure discovery — record the count of DD-eligible, same-unit nodes with no producing source, to decide whether the discovery axis is worth any further work.

§ Decisions

Is HAS_STANDARD_NAME authored truth or a derived projection?

LOCKED. The edge is structurally a 1-hop denormalization of the provenance 2-hop; the root-cause bug was maintaining it by hand. Reconcile it from the authority (symmetric with source_paths). Not dropped: it is load-bearing in dozens of internal queries (review-context enrichment, source eligibility, coverage, dedup, derived-parent admission) plus the MCP edge-walkers — repointing them all to the 2-hop is infeasible. Not read by the export/catalog SPA (those use PRODUCED_NAME directly), so this repair is a codex-internal concern only; the published catalog is unaffected.

Where does the backfill live?

LOCKED. Mirrors the existing scalar reconcile; self-heals every run so any later-composed source is picked up automatically. No migration script (banned).

What happens when a provenance pair's DD unit and SN unit disagree?

LOCKED. units_agree is the safety gate; violating it surfaces a real defect rather than propagating it into the index.

Is a true discovery axis (DD-eligible nodes with no producing source) worth building?

Open — gated on the measured residual from §3 step 4. Whatever the answer, no orphan HAS_STANDARD_NAME edge is authored without a source behind it.
