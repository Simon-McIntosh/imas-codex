§1 — The gap: an accepted name attaches only to where it was born

A StandardName is a concept — ion_atomic_number, electron_temperature, poloidal_magnetic_flux. The IMAS DD contains that same physical quantity at many paths across many IDSs (every z_n leaf is an atomic number; every Langmuir-probe t_e is an electron temperature). The (:IMASNode)-[:HAS_STANDARD_NAME]->(:StandardName) edge is what turns the standard-name vocabulary into a navigation index over the DD: given a name, list every DD path that realizes it; given a path, read its canonical name.

Today that index is almost empty. The generate pipeline writes a HAS_STANDARD_NAME edge only from the single source that seeded the name (plus whatever consolidation happened to touch), so the concept never reaches its siblings. The live graph:

Lineage. This plan is the sibling axis to sn-dd-unit-curation (followup f-ddhu-004 from the shipped dd-horizon-update). That plan builds the order-insensitive units_agree canonical-unit normalizer (W.m^-2 ≡ m^-2.W); this plan consumes it as the safety gate for every backfilled edge. It was originally sketched as f-sdu-005 inside the unit-curation campaign and is promoted here to its own plan.

§2 — Two mechanisms, one gate

The backfill has two distinct populations of missing edge, with very different risk:

  • Mechanism 1 — Realize (deterministic, $0). For every accepted name, source_paths already names DD paths that carry no HAS_STANDARD_NAME edge (3423 edges / 1522 names). These are already asserted attachments the write path dropped. Materialize them — but re-gate each entry through DD-eligibility (node_category ∈ SN_SOURCE_CATEGORIES) and units_agree, so a stale or wrong scalar entry is rejected rather than trusted blindly. No LLM, no review.
  • Mechanism 2 — Fan-out (discovery). Beyond what source_paths ever recorded, an accepted name should attach to every DD-eligible node that is the same quantity — the "ion_atomic_number → all z_n" case, including paths never seen during generation. This needs a positive match predicate, not just a unit check: units_agree alone would attach every dimensionless integer to ion_atomic_number. Candidate predicates are the open design question in §4.

units_agree is the shared floor for both: no edge is written unless the DD-path unit and the SN unit reconcile under the canonical-unit normalizer. An edge that would violate units_agree is a signal of a wrong name or a wrong source, not an edge to force.

§3 — Implementation approach (to be locked in §4)

The strong prior is a single idempotent reconcile sub-step that runs on every sn run post-drain (alongside the existing source-path / vocab-gap reconcile in loop.py), plus an sn run --only attach entry point for a one-shot backfill of the current backlog. Idempotency is mandatory — the SN pipeline reconciles continuously, so the pass must be safe to re-run and must converge (running twice writes zero new edges). It mirrors the existing reconcile_standard_name_sources / source-path reconcile pattern in graph_ops.py rather than adding a one-off migration script (banned per AGENTS.md).

Concretely, per accepted (and derived-parent-accepted) StandardName:

  1. Realize pass — MERGE an edge for each source_paths entry that resolves to a live, DD-eligible IMASNode whose HAS_UNIT target units_agree with sn.unit. Drop (and log) entries that fail either gate; feed genuine failures to the unit-curation triage.
  2. Fan-out pass — for each name, enumerate DD-eligible candidate nodes by the §4 match predicate, gate on units_agree, and attach. Emit a count of newly discovered edges per name so over-attachment is visible.
  3. Reconcile the scalar — after edges are authoritative, re-derive source_paths from the live edges (the existing scalar-from-edges reconcile), so scalar and graph agree in both directions.
  4. Guardrail — extend the SN-link integrity test: edges non-decreasing across a run, zero dangling, and every HAS_STANDARD_NAME edge satisfies units_agree. This is the acceptance oracle.

§ Decisions

What positive-match predicate gates Mechanism-2 fan-out (beyond units_agree)?

How do we prevent over-attachment for dimensionless / weak-unit concepts (unit = 1), where units_agree is nearly vacuous?

Where does the backfill live?

Does the Realize pass trust source_paths, or re-gate every entry?
