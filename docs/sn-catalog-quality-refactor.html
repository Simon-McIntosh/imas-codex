§1 — What has landed so far

This migration keeps the real state from plan 36 rather than its old “dispatch-ready” label. The surveyed implementation fraction is only 0.21: most of the progress so far is in design closure, audit reconciliation, and prompt/schema scoping, not in the final deployment loop. The important thing that has landed is that Phases 0–3 now have a stable specification and a reconciled backlog.

  • Phase 0 settled the grammar direction. The plan now consistently targets postfix composition, a closed reserved suffix vocabulary, vector/complex parent-part unification, and the audit/prompt changes needed to consume that grammar.
  • Phase 1 settled the catalog field cleanup. validity_domain, constraints, and catalog-facing cocos_transformation_type are designated for removal; doc_resolution_status is explicitly split from link_status; export exclusion rules for unresolved/internal links are defined.
  • Phase 2 settled the linking rebuild scope. The 11-bug failure chain is documented, the graph-backed context amplification path is specified, and the Delta A–J prompt/context expansions now cover cluster peers, hybrid DD search, identifier values, version history, and signal parity.
  • Phase 3 settled prompt and rubric hardening. Response-model cleanup, enrich-style-guide rewrites, range/unit validation, tokamak-example grounding, rubric recalibration, and the round-4 reconciliations (H-1, H-2, M-1, M-2, M-3, L-1) are all folded into the active plan.

§2 — What is still open

The remaining work is the deployment tail: proving the redesigned pipeline on live graph data, exporting only clean catalog entries, and closing the derived-parent lifecycle end to end.

  • Phase 4a–4c validation loop still has to run in reality. The plan requires a StandardName-only clear, a small equilibrium verifier run, structural gates for links/siblings/validators, and then domain-by-domain rotation under capped spend. None of that should be treated as already complete.
  • Phase 4d derived-parent bootstrap is still the deployment-critical gap. The renamed sn bootstrap-derived-parents workflow is meant to unify vector and complex parent recovery under postfix form, but the current code audit says the delete-on-failed-derived-parent path is not actually implemented yet.
  • Phase 5 export/publish remains open. The catalog handoff still has to prove zero exported dd: links, zero unresolved entries, satisfied parent/part invariants, and a clean ISN PR/rendering pass.
  • Phase 6 documentation is still open. AGENTS guidance, architecture docs, and the plan ledger all still need the postfix/linking/derived-parent model documented after code lands.

Blocking note from the survey: Phase 6’s “delete failed derived parents” expectation is not merely paperwork. It is currently a release blocker because the implementation stub remains in the codebase while the plan assumes the lifecycle already exists.

§3 — Code gaps the HTML plan must keep visible

  • Derived-parent deletion is still a stub. The prior audit flagged imas_codex/standard_names/graph_ops.py:7975 as the Phase 6 placeholder instead of a finished delete/rewire path.
  • proposed_verdict and specificity_score are not wired through runtime. The schema fields exist in the plan/design layer, but the survey found they are never meaningfully set or consumed by the running code.
  • Derived-parent specificity review is not actually active. The plan expects specificity-based rejection of weak derived parents; the runtime still lacks the corresponding review/persistence/deletion loop.
  • Plan 36 therefore cannot be treated as “ready to dispatch” without qualification. It is ready as a design package, but not as a statement that later lifecycle phases have already landed.

§4 — Open decisions

The markdown plan still leaves several implementation decisions unresolved. These should stay visible during the HTML migration so later workers do not mistake them for locked choices.

  • Should _amplitude remain scoped to kind=complex only, or is there any legitimate vector use that should be allowed alongside _magnitude?
  • Should documentation placeholder resolution remain parallel to link resolution (doc_resolution_status + link_status), or be collapsed into one shared status field?
  • Should the Phase 4 verifier gate stay structural only, or also require a reviewer-score threshold before the domain rotation opens up?
  • Should link ordering preserve emission order for provenance, or be canonicalized for stable export diffs?
  • Should complex-vector / complex-tensor remain deferred until first real need, or be predeclared up front?
