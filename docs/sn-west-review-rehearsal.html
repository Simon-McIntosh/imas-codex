§1 — Goal & scope

The approval pipeline is built and tested (sn-catalog-approval-pipeline, all six phases + the converged CLI). What has never happened is a single operational pass of the whole loop on real data with a human in the loop. This plan is that pass, in two acts:

  1. Dress rehearsal on the fork — the full mop-up → batch RC → PR → review → merge → fold-back → unwind cycle, with the PR raised within the fork (--target fork, the auto default for an RC batch) so gh, the review UI, and every human touch point are exercised with zero upstream noise. Everything is reversible: sn merge --undo + a graph snapshot.
  2. The real thing — the identical release command with --final, directing the PR at iterorganization/imas-standard-names-catalog for expert review. That merge is the SN set's first human validation; its approvals stay.

The corpus is the real WEST Task-2e set: 355 SN-eligible DD-v4 sources (imas_codex/standard_names/manifests/west_task_2e.yaml, shot 62253, 22 IDS). Stage 1 drains those sources through the paid LLM pipeline — this plan is therefore cost-gated and user-triggered; nothing here runs autonomously.

§2 — The converged CLI surface (record)

One batch token, three verbs — settled and shipped in the parent plan's follow-on work (commits c2983802, ecfb004b, 35230189):

  • sn run --batch west_task_2e — mop-up. Resolves the short name under standard_names/manifests/; gap-only by default (--reseed opts out). --focus remains for bare DD paths.
  • sn release --batch west_task_2e --bump minor -m "…" — the single orchestrator: mint → freeze artifact (manifests/reviews/<rc>.sn_names.yaml) → export approved ∪ batch → branch review/<rc> → push to the fork → gh pr create → back-fill PR number/URL. --target auto: RC batch → fork PR; --final → upstream PR (still an RC version — a batch never cuts a stable tag; --final only directs the PR). The branch always pushes to the fork; upstream main is never pushed directly. PR repo/owner derive from the ISNC checkout's own git remotes. The PR description is synthesized by the sn-release-notes seat from the release message + batch artifact + per-domain catalog diff (grounded; static fallback; --no-notes to skip).
  • sn merge --pr <url> — fold-back. The URL alone resolves the PR number, merge commit, diff base (<merge-commit>^1) and the frozen batch artifact (from the review/<rc> branch name). Untouched batch names auto-promote accepted→approved; edited names re-review (pass → approved, fail → contested). sn merge --undo --pr <url> unwinds the promotions (accepted human edits stay — graph history).

§3 — Rehearsal procedure (act 1: fork PR)

Full commands, in order — each stage records its outcome in this plan before the next runs:

  1. Snapshot. imas-codex graph export (restore path: graph load); optionally imas-codex graph push --dev for an offsite GHCR copy. Record: archive path.
  2. Mop-up (LLM spend — size the cap). uv run imas-codex sn run --batch west_task_2e --cost-limit <N> — gap-only, so re-runs only work the residual gap. Done-when: every eligible WEST source has an accepted, valid name; record the residual-gap count and skip reasons.
  3. Cut the batch RC (fork PR — the auto target for an RC batch). uv run imas-codex sn release --batch west_task_2e --bump minor -m "WEST review batch rehearsal" Done-when: additive diff (batch added, approved set byte-identical); catalog.yml carries export_scope: review + review_batch; artifact frozen and RC-tagged; PR open on the fork. Record: RC version, PR URL, batch size, the LLM-written PR body (judge its quality — this is the first live output of the sn-release-notes seat).
  4. Human touch points (the point of the rehearsal). Build/serve the RC data.json; confirm the SPA fixed batch view. In the gh UI: review the PR, comment, make one good edit and one deliberately non-compliant edit to two batch entries, merge. Record: friction notes — anything that felt awkward is a design finding.
  5. Fold back. uv run imas-codex sn merge --pr <merged-pr-url> Done-when: untouched names approved with provenance; the good edit re-reviewed → approved; the bad edit → contested (visible in sn status --contested). Exercise one resolution path (sn revert <id> --reason "rehearsal").
  6. Freeze check, then unwind. uv run imas-codex sn run (no scope) — no approved name mutates; contested/approved are pool-excluded. Then uv run imas-codex sn merge --undo --pr <merged-pr-url> — approvals drop back to accepted, provenance cleared. Record: undo counts match the merge counts.

§4 — Act 2: the real upstream PR

Only after §3 passes without design findings (or after fixing what it surfaces): 

uv run imas-codex sn release --batch west_task_2e --final -m "WEST Task-2e standard-name review batch"
# --final directs the PR at upstream (iterorganization/imas-standard-names-catalog);
# still an RC version; branch still pushes to the fork.
# … expert review happens upstream; on merge:
uv run imas-codex sn merge --pr <upstream-pr-url>      # approvals are FINAL — no undo

Deliverable: the WEST batch approved in the graph with full PR provenance — the machine-generated nameset's first human validation.

§5 — Graph-state management & cross-checks

  • State discipline: snapshot before act 1 (stage 0); the rehearsal's graph writes are cleared by --undo (stage 6), with graph load as the belt-and-suspenders fallback. Act 2 writes are permanent by design.
  • Backup policy (settled): the catalog is not the backup — it is lossy (no scores, chains, review history). The graph→GHCR archive (graph export / graph push --dev; automated by the imas-codex release CLI) is the backup. Review PRs stay approved ∪ batch, never full dumps.
  • Pre-flight cross-checks (things the implementation session flagged; verify before stage 2): the ISN dep pin in pyproject.toml can be bumped to 411d812 so the review export validates catalog.yml without a warning (the marker is written either way); confirm the scratch/fork ISNC checkout's origin/upstream remotes since they drive the PR target; confirm gh auth status on this host.
  • Known deferred items (out of scope here, tracked in the parent plan): the batching structure for future corpora; the one-time re-baseline of the published full catalog onto the approved-only model; the plain (non-batch) --final release restructure — content via fork→upstream PR with a tag-after-merge step, mirroring the batch flow — is an open decision below.
