§1 — Goal & context

We want the first human-review gate for the machine-generated standard-name (SN) set. The vehicle is a concrete, bounded corpus: the DD paths actually populated in the WEST production database (Task 2e DB analysis, shot 62253). The end-to-end intent is:

  1. Ensure every eligible WEST DD source (physical, geometric, coordinate quantities) carries an accepted standard name in the local graph — running focused mop-up where names are missing.
  2. Export only that subset of accepted names and open a pull request against the upstream catalog repository (ISNC) for human experts to review.
  3. On PR merge, flip those names' lifecycle to a frozen approved state that is immune to further mutation by general sn run pools.
  4. Import the approval back into the local graph — because the graph is the single source of truth and lives locally, the merged PR is the human gate but never itself writes the graph; a local import step always follows approval.

This plan is a design. It records what already exists (so we do not rebuild it), isolates the narrow gaps, and specifies the TDD and end-to-end verification for each. The concrete input corpus — imas_codex/standard_names/manifests/west_task_2e.yaml, 351 validated DD v4 sources with provenance, governed by the committed schema imas_codex/standard_names/config/sn_sources.schema.json — is already produced and committed; see §3.

§2 — Pipeline architecture

Two coupled loops. The local loop (graph = source of truth) mints accepted names from the manifest. The upstream loop is the human gate: a subset PR against ISNC, reviewed by experts, whose merge triggers a local import that freezes the approved names in the graph. A GitHub-hosted CI runner cannot reach the local Neo4j graph, so the authoritative graph write is always a local step — CI's role is to validate the PR and record the merge metadata (PR number, URL, merge-commit sha) that the local import consumes.

§3 — Input corpus & the sn-sources manifest schema

The WEST workbook lists 744 DD leaf paths across 22 IDS sheets, in DD v3-era spelling. Producing the corpus involved three transforms, all already done and captured in imas_codex/standard_names/manifests/west_task_2e.yaml:

  • Structural pre-filter — drop ids_properties/* and code/* (744 → 507).
  • DD v3 → v4 resolution — every path validated via the imas-dd MCP tools and the migration data; 40 renamed (e.g. b_field_tor_probe → b_field_phi_probe, j_tor → j_phi, bpol_probe → b_field_pol_probe, ECE t_e → t_radiation), 43 removed with no v4 equivalent (the entire boundary_separatrix subtree, boundary/x_point, boundary/strike_point). The resolution: block records every rename and removal so the conversion is auditable.
  • SN-eligibility filter — classify each resolved path by the graph's authoritative node_category, keeping only {quantity, geometry, coordinate} (imas_codex.core.node_categories.SN_SOURCE_CATEGORIES). Signal containers collapse to the nameable parent (magnetics/ip/data → magnetics/ip); depth-≥3 time leaves are skipped; structural, metadata, error, identifier, and fit nodes are excluded. Result: 351 distinct eligible v4 sources (after the classifier fix below).

Schema decision — yes, a light schema earns its keep. A bare path list cannot carry the three things this file must provide: (a) provenance so the v3→v4 conversion is reproducible and auditable, (b) a stable identity so the same file drives both the focus mop-up and the export-subset selection, and (c) a way to surface paths that need a human/classifier decision. The schema_version: 1 shape is: 

schema_version: 1
name: west-task-2e
title: ...
description: ...
provenance:
  facility: WEST
  source_document: research/west/Task_2e_WEST_DB_analysis_20251113.xlsx
  reference_shot: 62253
  dd_version_target: "4.0.0"
  eligibility_policy: sn_source_categories   # {quantity, geometry, coordinate}
  path_form: dd_source_id                    # signal '/data' leaves collapsed to parent
counts: { raw_leaves, after_structural_filter, eligible_sources, ... }
sources:                       # canonical, machine-consumable; loader flattens to <ids>/<path>
  <ids>: [ <ids-relative path>, ... ]
resolution:                    # informational audit of the v3→v4 conversion
  renamed: [ { from, to }, ... ]
  removed: [ { path, reason }, ... ]
review_candidates:             # dimensionless physical signals mis-gated as structural
  - { path, note }

The sources mapping is grouped by IDS for human readability; a loader flattens sources.<ids>[] to <ids>/<path>. Entries are DD source ids (the nameable node), not raw /data leaves — this matches how the graph stores a StandardNameSource and how export joins names back to paths. Full per-path exclusion detail lives in the west_sn_sources.exclusions.json sidecar to keep the driver file lean.

Classifier gap — found and fixed (landed). Producing this corpus surfaced a real DD-classifier defect: a whole group of dimensionless physical signals was classified structural and thus invisible to the extractor. Root cause: a signal is stored as a STRUCTURE container with a data leaf and the explicit dimensionless unit '1', but the container is classified while its unit is still unresolved at build time, so Rule 12 (STRUCTURE-without-unit → structural) demotes it — the same asymmetry that leaves dimensioned signals as quantity only when their unit happens to be present at pass 1. The fix is a pass-2 signal-signature promotion: a structural STRUCTURE/STRUCT_ARRAY carrying unit '1' with a direct data child is promoted to quantity (node_classifier.classify_node_pass2 R5; build_dd._reclassify_relational), matching the signal signature the extractor already honours. Scope is exactly the 23 genuine dimensionless quantities across the DD (Z_eff, optical depth, ellipticity, Mach number, flux expansion, beam fractions, strain, jₕ skew/kurtosis, …); the 148 '1'-unit containers without a data child (fit artifacts, pulse_schedule reference waveforms, summary structs whose value lives in /value) are deliberately left structural. Landed with unit tests (Red→Green) and applied to the live graph; the four WEST review-candidates now flow into sources, lifting the corpus from 346 to 351.

§4 — What already exists (build only the gaps)

A machinery scan of imas-codex and ISNC found that most of the target design is already scaffolded. This narrows the work sharply — we wire up and fill four gaps rather than build a pipeline. Grounding references:

  • The approved lifecycle state exists. NameStage in imas_codex/schemas/standard_name.yaml:305-343 is pending → drafted → reviewed → accepted → approved, with approved documented as "accepted through a merged catalog PR … set only when the PR number, URL and merge commit are recorded; RC export/import never grants it." That is exactly the frozen governance state we want.
  • The freeze exists. sn edit excludes approved from rename eligibility (edit.py:102) and from docs edits (edit.py:1019-1027); field-level protection.py strips protected editorial fields from pipeline writes for any approved / origin='catalog_edit' name (protection.py:119). The name-pipeline pools never re-claim approved/accepted/superseded/exhausted names.
  • Catalog-PR provenance fields exist on the schema (standard_name.yaml:481-512): catalog_pr_number, catalog_pr_url, catalog_approved_at, catalog_merge_commit_sha, exported_at, origin.
  • Export exists. sn release --export-only (cli/sn.py:3698-3841, core export.py:run_export) writes one YAML per physics domain plus a catalog.yml manifest — the exact live ISNC layout. Candidate predicate: name_stage IN ['accepted','approved'] AND validation_status='valid' AND docs_stage='accepted'.
  • Approve-on-merge exists. sn merge (cli/sn.py:4042, merge.py:run_merge) diffs a reviewed PR branch and, with --pr-number/--pr-url/--merge-commit, calls mark_catalog_name_approved (merge.py:653-681) to flip accepted→approved and record the provenance fields.
  • Import/reconcile exists. catalog_reconcile.reconcile_catalog (catalog_reconcile.py:137) diffs catalog YAML by id and updates editorial fields + provenance for matched names.

The genuine gaps, and only these, are the subject of §5–§8.

§5 — Gap A: focus mop-up from the manifest (--focus-file)

sn run --focus exists but takes bare paths on the command line only; there is no file input. Add sn run --focus-file <manifest> that loads a schema_version: 1 sn-sources file, flattens sources.<ids>[] to <ids>/<path>, and feeds the existing focus path list. This is a thin loader over the existing focus machinery.

Two behaviours to get right, both flagged by prior incidents. First, focus seeding mints StandardNameSource nodes directly and bypasses the eligibility gate at seed time — which is precisely why the manifest is pre-filtered to eligible sources in §3; the file is the gate. Second, focus is known to recompose existing names (near-duplicates) and to re-stage cross-linked accepted names. For a mop-up whose purpose is to fill gaps, not churn, the loader must default to seeding only paths that lack a live accepted name, and must not reset already-accepted names on those paths. This is the load-bearing design decision D-focus below.

§6 — Gap B: batch-scoped, additive export (never subset-replace)

Correction to an earlier design error. A first instinct was to export only the batch's names into the domain files. That is wrong: the catalog on ISNC main already holds the full accepted corpus (~2345 names, status: active), so writing only a subset would delete every other name from the files — a batch PR would read as "removes 2000 SNs." Diffs across batches must be strictly additive.

The rule: the export is always the full catalog, written deterministically (all names, stable sort by id, canonical field order). Because unchanged names serialise byte-identically, a re-export against main produces a diff of only what this batch changes — no phantom removals. The batch is the scoped subset whose entries actually change in this PR: they gain a review/approval stamp (§7) and any reviewer edits. So Gap B is not "export a subset" but "export the whole catalog with a scoped set marked" — the scope selector (--sources-file <manifest>, --domain, or an id list) picks which names carry the stamp, and the export records the batch's included-id set in the manifest so §8 approves exactly that set. Determinism (stable order + canonical serialisation) is the load-bearing property; without it, incidental reordering pollutes every diff.

§7 — Gap C: additive batch-review PR + how approval is recorded

Because the names are already published on main, a review PR does not add names — it elevates a batch to approved. To make that a clean, self-documenting diff, each batch's entries carry a visible review stamp in the catalog (e.g. a per-entry review: { pr, approved_at } block, or promotion of a reviewed status). A batch PR is then: full deterministic re-export with the batch's N entries gaining the stamp (± reviewer edits), on a branch review/<batch>, with a PR body listing the batch and its provenance. The diff shows exactly those N entries getting + review: next to their full definition — reviewers see the definition and the approval in one place, and merging is the approval. ISNC's existing validate.yml gates it; no new validation CI is needed.

Practical options for 2345 names. Two catalog models keep diffs additive: (a) in-catalog stamp (recommended) — approval recorded per entry as above; definition + approval reviewed together; costs one small ISNC schema addition. (b) separate append-only approved list — a approved/<domain>.yml that each batch appends ids to; catalog files untouched, diff is purely +ids, but reviewers must cross-reference definitions (mitigated by rendering them in the PR body). Either way the review corpus is chunked into human-sized batches: scope by (facility manifest) ∩ (physics domain or IDS), ~50–150 names per PR, so a domain expert reviews one coherent diagnostic/domain in one sitting. WEST (355) is the pilot facility, itself sub-batched by domain/IDS. Batches proceed monotonically — each merged batch stays approved and byte-identical in later diffs.

§8 — Gap D: import from the approved PR (not sn merge)

sn merge is the wrong tool for the human gate. It re-attaches the PR's diffed entries as steered edits and re-scores them through the LLM review, quarantining any that fall below threshold (merge.py:run_merge). That inverts the trust model: a human expert just approved these names, so re-running the machine reviewer — which can reject a human-approved name — is exactly backwards. It also only touches wording-diffed entries, so a batch approved as-is marks nothing.

The correct back-import treats the merged PR as authoritative. Call it sn import (from an approved/merged PR): given the merged branch + PR metadata (number, URL, merge-commit sha), it (1) applies any reviewer editorial edits via catalog_reconcile — scalar fields only, no re-scoring; and (2) marks the batch's names approved via mark_catalog_name_approved, recording the provenance fields. It never quarantines. The batch set comes from the review stamp / included-id set the export recorded (§6), so it approves exactly the merged batch, not just the diffed rows. catalog_reconcile already does the edit-apply half (it has no CLI today — give it one); the approval half reuses the existing per-name mark_catalog_name_approved. sn merge's re-score path is retained only for a genuinely-untrusted-edit scenario, if at all.

The CI-vs-local split is a hard constraint, not a choice. A GitHub-hosted runner has no route to the local Neo4j graph. So the "CI watcher that sets lifecycle to approved on merge" resolves to: ISNC CI validates the PR and, on merge, records the merge metadata (PR number, URL, merge-commit sha) as a durable signal (a workflow artifact, a release, or a committed marker); the authoritative graph write is a local sn import --pr-number … --pr-url … --merge-commit …, run by the maintainer or a local poller that watches ISNC for merged review PRs. The user noted this: "the graph is local so we will always need an import step following PR approval." The design honours it rather than pretending CI can close the loop.

§9 — TDD strategy

Each gap is specified test-first. Tests run against the live Neo4j (graph-marked tests are run, not skipped) per project convention, and pure logic is unit-tested with injected callables where the existing modules already allow it (e.g. campaign.py, merge.py use injectable graph callables).

  • Manifest loader (Gap A): a pure load_sources_file() test — a fixture YAML round-trips to the expected flattened <ids>/<path> set; rejects unknown schema_version; ignores review_candidates. Gap-only seeding: given a graph with some paths already accepted, the focus seed touches only the not-yet-named paths and does not reset accepted names (mock graph).
  • Subset export (Gap B): given a graph with names spanning several domains, --sources-file selects exactly the names whose source_paths intersect the manifest; the staged domain files contain only those; the export manifest records export_scope: manifest_subset and the included id set. Boundary cases: a name with multiple source paths (one in, one out) is included; a manifest path with no accepted name is reported, not silently dropped.
  • Subset PR (Gap C): the export-into-checkout produces a diff touching only subset entries; a name absent from ISNC is an addition, an existing name is an idempotent no-op. (PR creation itself is exercised via a dry-run / local bare-repo fixture, not a live GitHub call.)
  • Bulk approve-on-merge (Gap D): given a merged subset and PR metadata, every name in the subset ends name_stage='approved' with the four catalog-PR fields populated; names outside the subset are untouched; an entry whose wording the reviewers edited is re-scored before approval; approval is idempotent (re-running does not error or double-write). Guard test: without PR metadata, nothing is approved.
  • Freeze (Gap E, mostly verification): an approved name rejects sn edit rename and docs edits; a subsequent sn run pool pass does not re-claim it and protection.py strips editorial writes to it. These assert existing behaviour so a regression is caught.

§10 — End-to-end verification procedure

A single rehearsal against a throwaway ISNC checkout and the local graph, proving the loop closes before any real upstream PR:

  1. sn run --focus-file imas_codex/standard_names/manifests/west_task_2e.yaml → assert every eligible WEST source has an accepted, valid name (report the residual gaps and their skip reasons).
  2. Full deterministic export with the batch scoped (--sources-file …) into a scratch ISNC checkout → assert the diff is additive: only the batch's entries change (they gain the review stamp), every other name byte-identical, nothing removed.
  3. Open the PR against a fork/scratch ISNC; assert validate.yml passes and the diff is the batch only.
  4. Merge the scratch PR; run the local sn import --pr-number … --pr-url … --merge-commit … → assert every batch name is approved with provenance fields set, reviewer edits applied, and nothing re-scored or quarantined.
  5. Re-run sn run (no focus) → assert no approved WEST name mutated (frozen), and sn edit on one is rejected.

Only after this rehearsal passes do we run the real corpus and open the real ISNC PR for expert review — the deliverable that gives the SN set its first human validation.
