§1 — Goal & context

We want the first human-review gate for the machine-generated standard-name (SN) set. The vehicle is a concrete, bounded corpus: the DD paths actually populated in the WEST production database (Task 2e DB analysis, shot 62253). The end-to-end intent is:

  1. Ensure every eligible WEST DD source (physical, geometric, coordinate quantities) carries an accepted standard name in the local graph — running focused mop-up where names are missing.
  2. Export a small batch (carrying the already-approved set so the diff is additive) and open a pull request against the upstream catalog repository (ISNC) for human experts to review.
  3. On PR merge, promote the reviewed names' lifecycle to a frozen approved state immune to further mutation by general sn run pools — untouched names automatically, edited names only if they re-pass the compliance review.
  4. Bring the approval back into the local graph — because the graph is the single source of truth and lives locally, the merged PR is the human gate but never itself writes the graph; a local sn merge step always follows approval.

This plan is a design. It records what already exists (so we do not rebuild it), isolates the narrow gaps, and specifies the TDD and end-to-end verification for each. The concrete input corpus — imas_codex/standard_names/manifests/west_task_2e.yaml, 355 validated DD v4 sources with provenance, governed by the committed schema imas_codex/standard_names/config/sn_sources.schema.json — is already produced and committed; see §3.

§2 — Pipeline architecture

Two coupled loops. The local loop (graph = source of truth) mints accepted names from the manifest. The upstream loop is the human gate: an additive batch PR against ISNC, reviewed by experts, whose merge triggers a local sn merge that promotes the reviewed names to approved and freezes them in the graph. A GitHub-hosted CI runner cannot reach the local Neo4j graph, so the authoritative graph write is always a local step run by the maintainer — CI's only role is validate.yml on the PR. Review itself happens in the SPA (served from our ISNC fork), scoped to the PR's SNs and shared by URL (§7).

§3 — Input corpus & the sn-sources manifest schema

The WEST workbook lists 744 DD leaf paths across 22 IDS sheets, in DD v3-era spelling. Producing the corpus involved three transforms, all already done and captured in imas_codex/standard_names/manifests/west_task_2e.yaml:

  • Structural pre-filter — drop ids_properties/* and code/* (744 → 507).
  • DD v3 → v4 resolution — every path validated via the imas-dd MCP tools and the migration data; 61 renamed (e.g. b_field_tor_probe → b_field_phi_probe, j_tor → j_phi, bpol_probe → b_field_pol_probe, ECE t_e → t_radiation; the boundary_separatrix scalars collapse into boundary/*, resolved X-points map to contour_tree/node/{r,z}, and elongation_upper/lower both fold into the single boundary/elongation). Only 8 physical paths have no v4 target — active_limiter_point and strike_point (the latter still covered via the summary IDS) — plus a handful of never-eligible identifiers/flags. The resolution: block records every rename and removal so the conversion is auditable.
  • SN-eligibility filter — classify each resolved path by the graph's authoritative node_category, keeping only {quantity, geometry, coordinate} (imas_codex.core.node_categories.SN_SOURCE_CATEGORIES). Signal containers collapse to the nameable parent (magnetics/ip/data → magnetics/ip); depth-≥3 time leaves are skipped; structural, metadata, error, identifier, and fit nodes are excluded. Result: 355 distinct eligible v4 sources (after the classifier fix below and the boundary remap in the resolution record).

Schema decision — yes, a light schema earns its keep. A bare path list cannot carry the three things this file must provide: (a) provenance so the v3→v4 conversion is reproducible and auditable, (b) a stable identity so the same file drives both the focus mop-up and the export/approval selection, and (c) a way to surface paths that need a human/classifier decision. The schema_version: 1 shape is: 

schema_version: 1
name: west-task-2e
title: ...
description: ...
provenance:
  facility: WEST
  source_document: research/west/Task_2e_WEST_DB_analysis_20251113.xlsx
  reference_shot: 62253
  dd_version_target: "4.0.0"
  eligibility_policy: sn_source_categories   # {quantity, geometry, coordinate}
  path_form: dd_source_id                    # signal '/data' leaves collapsed to parent
counts: { raw_leaves, after_structural_filter, eligible_sources, ... }
sources:                       # canonical, machine-consumable; loader flattens to <ids>/<path>
  <ids>: [ <ids-relative path>, ... ]
resolution:                    # informational audit of the v3→v4 conversion
  renamed: [ { from, to }, ... ]
  removed: [ { path, reason }, ... ]
review_candidates:             # dimensionless physical signals mis-gated as structural
  - { path, note }

The sources mapping is grouped by IDS for human readability; a loader flattens sources.<ids>[] to <ids>/<path>. Entries are DD source ids (the nameable node), not raw /data leaves — this matches how the graph stores a StandardNameSource and how export joins names back to paths. Full per-path exclusion detail lives in the west_sn_sources.exclusions.json sidecar to keep the driver file lean.

Classifier gap — found and fixed (landed). Producing this corpus surfaced a real DD-classifier defect: a whole group of dimensionless physical signals was classified structural and thus invisible to the extractor. Root cause: a signal is stored as a STRUCTURE container with a data leaf and the explicit dimensionless unit '1', but the container is classified while its unit is still unresolved at build time, so Rule 12 (STRUCTURE-without-unit → structural) demotes it — the same asymmetry that leaves dimensioned signals as quantity only when their unit happens to be present at pass 1. The fix is a pass-2 signal-signature promotion: a structural STRUCTURE/STRUCT_ARRAY carrying unit '1' with a direct data child is promoted to quantity (node_classifier.classify_node_pass2 R5; build_dd._reclassify_relational), matching the signal signature the extractor already honours. Scope is exactly the 23 genuine dimensionless quantities across the DD (Z_eff, optical depth, ellipticity, Mach number, flux expansion, beam fractions, strain, jₕ skew/kurtosis, …); the 148 '1'-unit containers without a data child (fit artifacts, pulse_schedule reference waveforms, summary structs whose value lives in /value) are deliberately left structural. Landed with unit tests (Red→Green) and applied to the live graph; the four WEST review-candidates now flow into sources (the corpus is 355 after the boundary remap recorded in the resolution block).

§3b — Two manifest schemas & minting the SN set

The review operates on two different currencies, so it needs two file schemas:

  • sn-sources (sn_sources.schema.json, exists) — a set of DD paths (grouped by IDS). This is the extraction/mop-up currency: it drives which sources acquire names. The WEST manifest is one.
  • sn-names (sn_names.schema.json, to add) — a flat list of standard-name ids. This is the review/approval currency: export, PR, merge, and approval all operate on SN ids, not DD paths.

Minting the SN set from DD paths. The two are bridged by a deterministic join over graph state: mint_sn_list(dd_paths) = (1) base join — every StandardName whose source_paths list intersects the DD-path set (the link is the dd:<ids>/<path> URI on the SN, not a relationship); then (2) immediate-family closure — because approving a base without its siblings is incoherent, pull in only the immediate family of any name touched: its HAS_PARENT parent and that parent's direct children (siblings), one hop in each direction, non-transitive. Sharing a grammar base token (HAS_GEOMETRIC_BASE/HAS_SUBJECT) is deliberately not a closure edge — it is too broad — and neither is the transitive descendant tree, IN_CLUSTER, or REFINED_FROM lineage. This is the path→name join that the earlier design left unspecified — it is now the heart of the flow.

Do we need a fixed SN file? — derive live, but materialise for the record. Because the SN set is derivable, --focus <file> need not require a pre-made SN list: it dispatches on the file's schema — an sn-sources file is minted live to the SN set, an sn-names file is used directly. But the live mint depends on graph state at run time (which names exist, current family membership), so it is not reproducible across time. Therefore the release CLI should materialise the resolved SN list as an sn-names artifact and carry it with the batch — this frozen list is exactly the batch id-set carrier the earlier design lacked: it travels export → PR → merge so sn merge approves precisely the names that were reviewed, not a re-derived (possibly drifted) set. So: sn-sources is the human input; sn-names is the machine-frozen batch record; neither is mandatory as input, but the materialised sn-names list is the authoritative batch identity through the gate.

§6 — Gap B: export the growing approved catalog (small PRs, never the full corpus)

The ISNC review catalog is the approved nameset, and it builds up slowly. We do not dump the full accepted corpus (~2345 names) into a PR — from the start, and likely ever. Each --focus release exports the current approved nameset ∪ this focus batch, deterministically (stable sort by id, canonical field order). Since the already-approved names serialise byte-identically to what is on main, the PR diff is exactly the new batch — additive, no phantom removals. Early PRs are small (approved set ≈ empty, so the diff is just the batch); the approved catalog grows monotonically as batches merge.

So the export candidate predicate is name_stage == 'approved' OR id ∈ batch, not the current name_stage IN ['accepted','approved'] full dump. The batch is the minted SN set (§3b): the release CLI takes a focus input — an sn-sources file, minted live to the SN set, or an sn-names file used directly — resolves it to SN ids, and materialises that resolved set as an sn-names artifact. That frozen artifact is the batch's identity from here on: it is committed with the export and carried into the PR so §8's merge approves exactly it. Determinism (stable sort by id + canonical serialisation) is the load-bearing property — without it, incidental reordering pollutes every diff.

Where the frozen artifact lives & how it is traced (D-artifact-cli). The sn-names records are committed in imas-codex history under imas_codex/standard_names/manifests/reviews/<rc-version>.sn_names.yaml (beside the sn-sources manifests) — durable, reviewable, part of the repo's record. They are tagged by the RC version (e.g. v0.2.0rc65), which we assign ourselves and therefore know before any push — so traceability never depends on a PR number. A PR number cannot be reserved ahead of creation (GitHub assigns it on gh pr create), so it is back-filled: the artifact carries rc_version, minted_from (the sn-sources file), and the SN id list up front, and pr_number/pr_url/merge_commit are written in after the PR exists.

Fold it into one command so mistakes can't happen. Mint → freeze artifact (RC-tagged) → export the RC catalog (approved ∪ batch with review_batch stamped, §7) → bump versions.json → commit on a review branch → push to the fork → gh pr create (capture PR#/URL) → back-fill the artifact → commit + push must be a single sn release (review-batch) invocation. Doing these by hand risks partial/inconsistent states — an artifact that disagrees with the pushed catalog, a PR with no recorded batch, a forgotten versions.json bump. One orchestrating CLI keeps the artifact, the catalog, and the PR provably in lock-step.

Two questions are deferred to a future session (not needed to start): the catalog presently on main holds the full accepted set at status: active, so adopting the approved-only-growing model needs a one-time re-baseline (or a parallel approved catalog); and the batching structure itself — a focus manifest scopes the batch for now.

§7 — Gap C: the review PR and a PR-scoped SPA view

The additive PR (data). A batch's names are new additions to the approved catalog: the PR shows each batch name's full definition being added, on a branch review/<batch>, carrying the already-approved names unchanged so nothing reads as removed. Merging is the human approval act; §8 mirrors it into the graph. ISNC's existing validate.yml gates it — no new validation CI needed.

The existing SPA is the review surface — we do not build a new one. The catalog SPA (code in the ISN repo, site/src) already renders the catalog and already has a multi-release version selector: it deploys per version, each version folder carrying its own data.json (built by imas_standard_names/catalog/dataset.py) with a top-level versions.json driving the dropdown (the rc60…rc64 selector in the live UI). Our fork publishes RCs the same way, so each batch is just an RC = a catalog version, browsable and shareable at its version URL (…/imas-standard-names-catalog/<version>/#/browse/…) with no new app or hosting.

PR-scoping = a catalog-shape marker + an ISN-side filter. "Show only the items in this PR" resolves to: the review RC's catalog carries the batch's SN id-set (the materialised sn_names artifact, §3b), and the SPA, seeing a batch marker, auto-scopes to those names (with a toggle to reveal the already-approved context). Concretely, two small pieces:

  • Driving catalog shape (imas-codex export → dataset.py): emit the batch id-set into the RC's dataset — a top-level review_batch: [<sn-id>, …] (equivalently the sn_names list). This is the "correct shape" requirement: the export must stamp the batch into data.json. Normal (non-review) builds omit it.
  • Filter (ISN repo SPA): when data.json carries review_batch, the SPA renders a fixed view of exactly those names — no "show all" toggle; the PR view is the batch. It reuses the existing Filters/ResultsList machinery (the batch set constrains the result universe), a small change in site/src, not a new SPA. The already-approved names remain browsable as normal via the version selector (a non-batch version shows the full catalog); the batch RC simply shows only its batch.

Deep links — advice: reuse what exists, add nothing new. The SPA already encodes state in the URL (url-state.js: #/<view>/<name>?q=…) and the version is already in the path, so the RC version URL is the shareable per-batch link — fixed to the batch because the catalog itself carries the marker. No PR-number routing, no bespoke deep-link machinery, and no scope toggle/param (the batch RC shows only its own names). The RC version URL is the whole story. Recommendation: ship the review_batch marker + fixed scope; add nothing else.

§8 — Gap D: sn merge from an accepted PR — the approval state machine

sn merge is the correct tool, and it runs only from an accepted (merged) PR — one with a merge commit. Per batch name in the PR:

  • Untouched (name and docs identical to the graph's version): the human approved it by merging, so it is promoted accepted → approved automatically — human approval plus an implicit LLM rubber-stamp / compliance pass. No re-review is spent on unchanged names.
  • Edited (name or docs changed by a reviewer): the edit re-triggers the full review — this is vital and non-negotiable, so no human wording change reaches the catalog without passing the pipeline's compliance rubric. If the edited form passes → approved. If it fails → the new contested state below.

New lifecycle state — contested. A name that was changed during review but whose edited form failed the compliance re-review lands in contested: neither approved (it failed the rubric) nor silently reverted (a human deliberately changed it). It is a human-vs-machine conflict that needs adjudication, so it is frozen and excluded from all pools (like exhausted/superseded) and surfaced in a report. Resolution paths: (a) a maintainer revises the wording via sn edit → re-review → approved; (b) an explicit override-approve (accept the human wording over the rubric, recording a justification) → approved; or (c) revert to the prior accepted form → accepted, re-opening it for a later review batch. The merged PR already records the human intent; contested captures only that the machine could not rubber-stamp that specific edit. The full transition set: accepted →(untouched, merged)→ approved; accepted →(edited)→ re-review → approved | contested; contested →(edit / override / revert)→ approved | accepted.

This mostly reuses existing machinery — read_pr_changes already separates edited from unchanged, the re-review/quarantine path exists in run_merge, and mark_catalog_name_approved sets the governance fields. The additions are: auto-approving the untouched batch (merge today only touches diffed rows), and the contested stage below.

contested — concrete spec. Worked out against the real schema and worker machinery:

  • Schema (imas_codex/schemas/standard_name.yaml, NameStage enum ~305-343): add contested alongside the other non-progressing states (refining/exhausted/superseded). Description: "Edited during human catalog review, but the edited form failed the compliance re-review; frozen pending human adjudication. Not published; not silently reverted." It is orthogonal to validation_status — a contested name simply never reaches the approved catalog because it is not approved, so no validation_status change is needed (leave it as it was; the name_stage gate does the work).
  • Freeze / pool-exclusion: the pool claim predicates in workers.py that already exclude ['superseded','exhausted'] (and never re-claim accepted/approved) gain contested in the exclusion set — one predicate change per claim query — so no generate/review/docs pool ever touches it. It is thus as frozen as approved, but unlike approved it is neither published nor terminal — it is a holding state.
  • Entry point: set only by sn merge on an accepted PR, when a reviewer-edited name fails the re-review (the branch that today would quarantine). Instead of quarantine-on-an-accepted-name, the name moves accepted → contested with a StandardNameChange recording the failing edit and the rubric verdict.
  • Resolution (three explicit exits, all human-driven): (a) fix — sn edit revises the wording → the existing edit→review path → approved if it now passes, else back to contested; (b) override — a new sn approve --override <id> --reason "…" that force-sets approved over the rubric, recording the justification in the change ledger (human authority beats the machine rubric, but only deliberately and on the record); (c) revert — drop the reviewer edit back to the prior accepted form, returning the name to a future review batch unchanged.
  • Visibility: a sn status --contested report (or a facet in the pool-status output) lists contested names with their failing edit and verdict, so conflicts are actioned rather than lost. Because it is pool-excluded and unpublished, a contested name is inert until a human resolves it.

No CI signal/watcher — the merge step is a manual local run. The graph is local and a GitHub runner cannot reach it, so a CI job could not actually promote anything; there is little value in a watcher emitting a signal. The maintainer simply observes the merged PR (its number is right there) and runs the local sn merge --pr-number … --pr-url … --merge-commit … against the sn_names batch artifact. CI's only role stays what it already is — validate.yml on the PR. "The graph is local so we will always need an import step following PR approval," and that step is a deliberate local action, not automation.

§9 — TDD strategy

Each gap is specified test-first. Tests run against the live Neo4j (graph-marked tests are run, not skipped) per project convention, and pure logic is unit-tested with injected callables where the existing modules already allow it (e.g. campaign.py, merge.py use injectable graph callables).

  • Manifest loader (Gap A, SHIPPED): pure loader tests (done) — a fixture sn-sources YAML flattens to the expected <ids>/<path> set; unknown schema_version, mis-shaped sources, and non-YAML raise; file vs bare-path mixing dedups. Owed: a CLI-level test of the sn run --focus <file> wiring (currently only the pure loader is covered), and — once D-focus locks — a gap-only-seeding test proving accepted names on listed paths are not re-staged.
  • Minting (Gap B): mint_sn_list(dd_paths) is pure over an injected graph view — assert the base join (SN whose source_paths intersect the set, incl. a name with one in / one out path) and the family closure (a touched base pulls its immediate family — HAS_PARENT parent + that parent's direct children (siblings), one hop, non-transitive, and NOT shared-grammar-base names); a DD path with no linked SN is reported, not dropped.
  • Additive export (Gap B): export of approved ∪ batch is deterministic (stable sort + canonical serialisation) and, against an approved-only baseline, the diff is exactly the batch's additions — nothing else changes or is removed; the resolved sn-names artifact is materialised and matches the minted set.
  • Review PR (Gap C): the export-into-checkout yields an additive diff (batch definitions added, approved set byte-identical); PR creation exercised via a local bare-repo fixture, not a live GitHub call.
  • Merge state machine (Gap D): from a merged PR + the sn-names batch artifact — an untouched batch name auto-promotes to approved with provenance set (no re-review); an edited name is re-reviewed (pass → approved); an edit crafted to fail the rubric lands in contested; names outside the batch are untouched; without PR metadata nothing is approved; re-running is idempotent.
  • Freeze & contested (verification + new): an approved name rejects sn edit rename/docs edits, no pool re-claims it, and protection.py strips editorial writes (existing behaviour — regression guard); a contested name is likewise pool-excluded and is resolvable by edit→re-review, override-approve, or revert-to-accepted (new).

§10 — End-to-end verification procedure

A single rehearsal against a throwaway ISNC checkout and the local graph, proving the loop closes before any real upstream PR:

  1. sn run --focus imas_codex/standard_names/manifests/west_task_2e.yaml → assert every eligible WEST source has an accepted, valid name (report the residual gaps and their skip reasons).
  2. Deterministic export of approved ∪ batch into a scratch ISNC checkout → assert the diff is additive: the batch's full definitions are added, every already-approved name byte-identical, nothing removed.
  3. Open the PR against a fork/scratch ISNC; assert validate.yml passes and the diff is the batch only.
  4. Merge the scratch PR; run the local sn merge --pr-number … --pr-url … --merge-commit … → assert untouched batch names are approved with provenance set (no re-review), an edited name is re-reviewed (pass → approved), and an edit crafted to fail the rubric lands in contested (not approved, not silently reverted).
  5. Re-run sn run (no focus) → assert no approved name mutated (frozen), sn edit on one is rejected, and contested names are not claimed by any pool.

Only after this rehearsal passes do we run the real corpus and open the real ISNC PR for expert review — the deliverable that gives the SN set its first human validation.
