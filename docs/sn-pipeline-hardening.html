§1 — Why this plan exists

This plan captures the remaining work to make the IMAS Standard-Names (SN) generation pipeline trustworthy enough for a full from-scratch catalog regen. It was opened after a session that overturned a false conclusion: a physics-correctness benchmark had reported that every composer model (free local DeepSeek V4-flash and every frontier model) was collapsing locus/projection quantities to bare bases (e.g. the X-point R and the magnetic-axis R both named major_radius), which had been read as a model/prompt failure. That was a measurement artifact — the benchmark's own _resolve_name hand-rolled the IR and silently dropped the locus/projection segments before scoring. The models were emitting fully faithful IR all along.

The lesson that governs every task below: never trust "we inject or produce X" without exercising the real code path. That mistake bit three times in one session (the resolver, a doc-render check rendered at the wrong pipeline stage, and an audit that over-reported description truncation from its own print buffer). Every change in this plan must be verified on the real path, and every naming change must round-trip through ISN parse/compose.

§3 — The phase plan

Six phases, ordered by dependency. Phase 1 is a genuine correctness bug and gates a clean A/B, so it runs first; the from-scratch regen (Phase 6) is the endgame once Phases 1–5 are trustworthy.

Phase 1 — to_ir transformation routing (the real C-4 bug)

Problem. The compose model outputs grammar segments (operator_token, operator_kind), not a name string; to_ir()+compose() do the joining. But GrammarSegments.to_ir() (imas_codex/standard_names/models.py ~284-291) routes operator_token into OperatorApplication(kind=unary_prefix), and compose() joins all unary-prefix operators with _of_ → volume_averaged_of_electron_density, which fails to parse / round-trip (ValueError: projection lost token {'of'}). The canonical ISN IR puts these in the transformation field: the Transformation enum (~50 members) renders each correctly — averaging/reduction/normalization (volume_averaged, line_averaged, flux_surface_averaged, normalized) bare; differential/scope (time_derivative, gradient, radial_derivative) with _of_; and postfix (magnitude, real_part, imaginary_part, amplitude) as a suffix.

Deliverable. to_ir() routes operator_token to the transformation field (mapping to the right enum member, honouring operator_kind for the prefix/postfix variants such as real_part vs real_part_postfix), with a fallback to OperatorApplication only for any non-transformation operator. Mirror how imas_standard_names.grammar.parser builds the transformation IR (a naive StandardNameIR(transformation=…) construction raised a ValidationError in a probe — study the parser and match its construction). Also reconcile validate_candidate.

Done-when. A comprehensive round-trip test covers every operator class (averaging-bare, differential-_of_, postfix) and passes; the sn-compose stale shared-include examples are corrected (_grammar_reference.md:110,127 and _exemplars.md:28,34,52 still instruct the invalid volume_averaged_of_X form); the physics benchmark is re-run and is unchanged-or-improved vs the baseline. High blast radius — to_ir is on the path of every composed name, so the round-trip suite is mandatory before commit.

Phase 2 — Prompt consolidation (field-choice oriented)

Reframe (locked). The model returns segment candidates, and compose() assembles the name. So prompt rules that teach joining-word surface syntax (operator _of_ scope, _at_/_of_ spelling, adjacent-duplicate-token checks) teach a string the model never emits. These are noise to cut. Genuine field-choice guidance stays and is reframed as such: which locus_relation (of/at/over), which locus_token, which qualifiers, prefix-vs-postfix operator_kind, projection_shape.

Deliverable. A leaner generate_name_system.md (the audit measured ~1700 lines across prompt + includes; target a goal-first structure with one canonical statement of each rule). Lift the goal + 3 gold exemplars to the top. Fold the 6× _at_/_of_ and 5× projection restatements into one each, preserving the union of every nuance. Resolve the remaining dedup (the 11 HARD checks ~70% duplicate the 36 numbered rules; Rule 25 is a "same as Rule 19" stub).

Done-when (user mandate). Every removal is individually justified as pure duplication or contradiction — no hard-won signal is dropped (much was earned over many rotation cycles). Each change is A/B-measured against the baseline (physics_bench-20260616T100837Z) and stays ≥ baseline. The exemplars are corrected (no taught-then-banned forms).

Phase 3 — Compose grounding on per-member dd_paths

Problem. A compose candidate can cover N sibling DD paths (item.dd_paths), but _enrich_batch_items enriches only the primary path; the other members' DD documentation is never fetched or rendered, so a name spanning N leaves is grounded on one. (Lower severity than Phases 1–2: the input item already carries some sibling context via sibling_fields/cluster_siblings — this is a coverage enhancement, not a silent-drop bug.)

Deliverable. A compact dd_paths_docs {path: terse_dd_documentation} map built in _enrich_batch_items (single UNWIND query) + a guarded loop in the compose user template. Must follow Phase 2's reframe (terse DD doc, distinct from the rich description). Done-when: grouped names show each member's DD clause; A/B neutral-or-better.

Phase 4 — Docs-model re-assessment (DSv4 vs sonnet)

Context. The Phase-2 grounding fixes (closed-vocab block now renders for docs; chain_history wired; both description and DD doc served) materially improved docs grounding. [sn-docs] is currently sonnet-4.6; the pre-local-model best was gpt-5.5.

Deliverable. An A/B of DSv4-flash vs sonnet-4.6 on docs generation physics-correctness, using the benchmark's docs path (--include-docs + the description-rescore + docs review) and a docs gold standard. Re-use the calibrated physics judge adapted for docs prose. Done-when: a keep/switch decision for [sn-docs], backed by numbers — can it move to free DSv4, or does it stay sonnet-class?

Phase 5 — Admit signal-STRUCTURE SN sources (336 quantities)

Problem. Signal-pattern IMAS quantities (magnetics/ip, rogowski_coil/current, tf/coil/current, langmuir, ece, neutron_diagnostic, …) are wrongly excluded as SN sources by two gates: the extraction leaf-invariant (dd.py:524, NOT data_type IN [STRUCTURE,STRUCT_ARRAY]) and a latent qualify_dd S0 bug (value_type.startswith("STR") over-matches STRUCT_ARRAY/STRUCTURE). The classifier is correct (Rule 11b marks STRUCTURE+unit → quantity); the extraction gates override it.

Scope (locked). Admit the clean signal signature only: STRUCTURE/STRUCT_ARRAY quantity nodes with a /data child and no quantity child = 336 nodes across magnetics, langmuir_probes, divertors, neutron_diagnostic, ece, charge_exchange, ic_antennas, tf, pf_active, calorimetry. Do not admit the broader 447 "no quantity child" set (pulse_schedule setpoint refs, *_fit containers, summary species aggregates — separate classifier decisions). Done-when: the 336 are nameable; round-trip + the qualify_dd S0 prefix bug fixed; land before Phase 6 (it changes SN-source eligibility, so the regen must include them).

Phase 6 — Incremental rollout → scope-gated regen (endgame)

Reframed (lead, 2026-06-17): NOT a big-bang clear+regen. The pipeline is now correct (grammar rc43, consolidated prompt, device-handling, 336 signal sources) and the compose gate passes (physics 0.889 ≥ 0.865). The endgame is incremental:

  1. Restart --focus rotations on representative paths across domains to confirm the full pipeline works end-to-end with all fixes (compose/parse round-trip, units injection, review, persist).
  2. Progressively extend names-only generation, one physics domain at a time — record per-domain coverage + quality; do not clear the catalog.
  3. Pause before any clear+regen until the regen-scope decision is locked: names-only vs names+docs, and extracted-paths-only vs full-DD.

Tracked by f-snph-007 (rollout + item B drift-guard allowlist + item C gold regen + scope resolution). The full from-scratch clear+regen happens only after the scope decision; budget tracked via SNRun.cost_spent.

§ Decisions

Which model composes standard names?

LOCKED. Keep the free local hosted_vllm/deepseek-v4-flash . True baseline: DSv4 89% physics / 4-of-5 hard at $0 vs opus-4.8 100% / 6-6 at $1.15. DSv4 is competitive; the gap is one defensible specificity case. The earlier "switch composer" pressure came from the resolver artifact, now refuted.

Bare radius vs major_radius ?

LOCKED. major_radius = cylindrical R coordinate; minor_radius = flux-surface label; bare radius = an object's physical size (coil/fibre/curvature) — three distinct concepts sharing unit m (102 R-coordinate leaves, 59 size-radius leaves, ~20 minor). The qualifier is distinguishing, not a reflexive wart; keep it.

C-1: locus for shape parameters (triangularity, elongation)?

LOCKED. Shape parameters always name the surface they describe: _of_plasma_boundary for the LCFS contour, _of_flux_surface (or the specific interior surface) otherwise — we measure triangularity of interior surfaces too. Never bare. Both forms round-trip; this is convention. Applied in 61221460 .

C-4: operator joining — bare vs _of_ , and at which layer?

LOCKED. The bare averaging exemplar is correct; the _of_ form is invalid. The model outputs the operator segment and to_ir / compose must produce the right joining — it is NOT a prompt-surface rule. Fix is Phase 1 ( to_ir transformation routing). The _at_ / _of_ locus relations are distinct grammar segments the parser relies on — keep them.

C-/Phase-5: how wide to admit signal-STRUCTURE SN sources?

LOCKED. Admit only the unambiguous signal signature (STRUCTURE/STRUCT_ARRAY quantity with a /data child, no quantity child) = 336 nodes. Defer pulse_schedule references, *_fit containers, summary species aggregates as separate classifier decisions. Also fix the qualify_dd S0 startswith("STR") over-match.

What is the acceptance gate for any grammar/naming change?

LOCKED. Every change must round-trip through ISN parse / compose ; update compose / parse (or the codex to_ir mapping) as required. Exercise the real code path — do not assert behaviour from a bypassed stage.

OPEN (Phase 4): does [sn-docs] move to free DSv4-flash, or stay sonnet-class?

RESOLVED by the Phase-4 docs A/B/C (2026-06-16, 12-path set, opus-4.8 judge, post Option-5/resolver/grounding fixes). Ranking sonnet ≳ gpt-5.5 ≫ DSv4: sonnet-4.6 docs_quality 0.903 / physics_acc 17.8 / doc_qual 18.2 / 11 outstanding,1 good,0 inadequate ($0.79); gpt-5.5 0.882 / 17.3 / 17.9 / 10 outstanding,2 good,0 inadequate ($0.84); DSv4-flash 0.838 / 15.8 / 16.8 / 8 outstanding,2 good,2 INADEQUATE ($0). gpt-5.5 is a clean close second but costs MORE than sonnet for slightly lower quality, so it does not displace sonnet. DSv4's 2 inadequate docs disqualify the free option for human-facing documentation prose. Keep sonnet-4.6. Artifacts: research/docs_ab.json (sonnet+DSv4), research/docs_ab_gpt55.json (gpt-5.5).

Phase 6 regen scope: names-only vs names+docs, and extracted-paths vs full-DD

OPEN — resolve before any clear+regen. Two orthogonal axes: (a) names-only or names+docs; (b) only the already-extracted StandardNameSource paths or the full DD. Lead direction (2026-06-17): do NOT big-bang the clear+regen. Start incremental — restart --focus rotations to validate the full pipeline end-to-end, then extend names-only physics-domain-by-domain, and lock this scope before clearing the catalog.

ISN operator development: change_in operator + operator×projection coexistence

ISN investigation (2026-06-17) found two grammar gaps the compose prompt depends on. THREE prompt rules were already ISN-native and fixed in codex 086ae7e8 (region→over_region, ratio→ratio_of, outline_point→plasma_boundary).
