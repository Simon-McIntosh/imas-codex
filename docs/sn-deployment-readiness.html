SN Deployment Readiness Study — 2026-06

This research note is a deployment decision document, not a feature plan. It combines a fresh code audit of imas_codex/standard_names/ with the audited StandardName graph snapshot supplied from 2026-06-04. Where useful, I also note a 2026-06-05 authoring-time spot check against the live graph head; those head metrics are explicitly marked because the graph is still moving.

§1 — Executive summary & verdict

Verdict: CONDITIONAL-GO. The StandardName system is no longer an experimental stub: the six-pool pipeline runs, the reviewer chain works, documentation quality is high, and the catalog round-trip exists in code. The release should still be held behind three explicit conditions: derived-parent cleanup, resolution of the accepted low-score names, and either implementation or formal deferment of the long-promised “Phase 6” deletion path.

The evidence base is asymmetric. The quality signal is strong: in the frozen 2026-06-04 audit snapshot, 123 of 155 accepted names (79%) scored at least 0.90 and the docs score distribution was excellent (min 0.81, mean 0.92, max 0.99). The release risk is concentrated instead in lifecycle debt: five accepted names sat below 0.50, 26 names were exhausted, and the code still contains a comment-level promise that failed derived parents will be deleted later even though that delete path does not exist.

  • Condition 1: clean up the derived-parent tail, especially total_plasma_current and the other structural orphans.
  • Condition 2: resolve the accepted low-rsn names before a production catalog freeze.
  • Condition 3: either land the Phase 6 delete cascade or explicitly declare that failed derived parents are a manual curation responsibility.

§3 — Open work blocking production release

3a. BLOCKING — Phase 6 derived-parent deletion is not implemented

The most important release blocker is still structural, not statistical. claim_refine_name_batch() explicitly skips derived parents and documents the rationale in a comment: graph_ops.py:7972-7977 says the admission gate filters bad derived names at write time and “Phase 6 will delete failed ones explicitly via the dead-link cascade.” That delete cascade is not present. The matching schema fields — proposed_verdict and specificity_score in schemas/standard_name.yaml — remain advisory and inert.

  • What this means in practice: structurally redundant derived parents can survive in accepted state with low scores because the cleanup stage they depend on never landed.
  • Where it shows up: total_plasma_current is the clearest example in the supplied audit snapshot; it is a derived orphan with no credible reason to remain a catalog-grade name.
  • Release risk: the public catalog risks carrying redundancy that the codebase itself already knows is weak.
  • Required fix: either implement the delete cascade or make a deliberate product decision that weak derived parents are a manual curation problem rather than a runtime lifecycle state.

3b. IMPORTANT — accepted names with very low rsn still need intervention

The frozen study snapshot flagged five accepted names at rsn = 0.30 via the semantic-similarity gate: beta, average_magnetic_field, flux_surface_mean_magnetic_field, total_plasma_current, and average_magnetic_flux_due_to_external_coil. That is a tiny fraction of the catalog, but it is too visible a tail to ignore before a production release.

  • total_plasma_current is the strongest prune candidate because it is structural, derived, and redundant with the more stable plasma_current hierarchy.
  • beta is the hardest case because it is a legitimate physics quantity; this looks more like a scoring or description-quality mismatch than a bogus name.
  • The remaining names should go back through a targeted reset/re-review path rather than shipping as-is.

The pragmatic release action is straightforward: prune the obvious structural orphan, explicitly re-review beta, and run a low-score reset for the rest. What matters is not perfection; it is demonstrating that there is an active cleanup path instead of accepting these as harmless noise.

3c. IMPORTANT — 26 exhausted names need an explicit policy

The exhausted set is not a release blocker because exhausted names are not destined for the catalog by default. It is still a release concern because exhausted names encode unresolved pipeline debt: they are names that already consumed the allowed refinement budget and still failed acceptance. A production release should not proceed without deciding whether those 26 names are (a) correct failures that should be deleted or left internal forever, or (b) false negatives that deserve manual rescue.

  • If they are genuine failures, treat them as curation debt and prune or quarantine them explicitly.
  • If they are recoverable, write down the rescue rule now instead of leaving them as an indefinite tail.

3d. ADVISORY — export gate A is still a subprocess stub

export.py:_run_gate_a() still shells out to uv run pytest -m "graph or corpus_health" and labels itself a stub that “in Phase 6 … will be made more granular.” This does not invalidate release readiness, but it does mean the export gate is more brittle and less introspectable than the rest of the pipeline.

The key judgment call is therefore: do not let this advisory stub distract from the real blocker. The release risk is the missing derived-parent cleanup, not the fact that the gate is currently coarse-grained.

§5 — Recommended actions before v1 release

  1. Prune total_plasma_current. Owner: SN maintainer. Effort: ~5 minutes. This is the cleanest obvious orphan and the quickest way to prove the cleanup policy is real.
  2. Re-review beta. Owner: SN maintainer. Effort: one focused pipeline run, roughly $0.05. This is the most legitimate low-score name and the best candidate to separate scoring noise from real naming failure.
  3. Implement or explicitly gate Phase 6 derived-parent deletion. Owner: core SN developer. Effort: 1–2 days. The release should not go out while the code still promises a future delete cascade that does not exist.
  4. Audit the 26 exhausted names. Owner: SN maintainer. Effort: 1–2 hours. Decide whether they are correct failures to prune/document or rescue candidates that deserve manual intervention.
  5. Refine the remaining accepted names with rsn < 0.80 that are not structural parents. Owner: SN maintainer. Effort: small bounded rotation. The objective is to narrow the catalog tail, not to restart the pipeline.
  6. Correct plan inventory metadata on the six stale/unmigrated plans. Owner: docs/planning maintainer. Effort: 1–2 hours. The implementation matrix above shows that plan state is now lagging code state badly enough to confuse deployment decisions.

§7 — Final decision statement

The StandardName program has crossed the threshold from “experimental subsystem” to “production-capable subsystem with known cleanup debt.” That is why the correct verdict is CONDITIONAL-GO, not NO-GO. The system already does the hard part — generate, review, refine, document, export, and search names at a quality level that looks like a real catalog.

The only honest reason to withhold a production release is the unresolved lifecycle contradiction around derived parents. Fix that contradiction, resolve the obvious low-score tail, and the remaining open plans become ordinary enhancement backlog rather than release blockers.
