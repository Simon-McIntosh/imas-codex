Fix = (1) clean the graph cruft via a re-runnable code path that clears embedding_text/embedding_hash on non-embeddable nodes with no vector (the embed-category exclusion is intentional for search-noise reduction, so text-gen legitimately follows it); (2) category-scope test_no_embeddings_without_embedding to EMBEDDABLE_CATEGORIES (the real embedding contract, defensive against future reclassification cruft); (3) the 198 vector-but-no-text are all identifiers hit by a real persistence bug in embed_identifier_nodes (computes text, never SETs it) — fix the SET + backfill embedding_text on the 198 by regenerating deterministically from their descriptions (vector unchanged, $0).

Axis C — clear the COCOS half-state, or scope the test to current nodes?

All 20 half-state nodes are lifecycle_status='removed'. Root cause is build_dd.py's stale-label clear pass (~:1974): it sets cocos_transformation_type=null on stale paths but leaves cocos_label_source set. Groom fix = clear cocos_label_source in the same SET so removed nodes never enter a half-state (prevents recurrence on every build), then groom the existing 20 by applying the corrected clear logic. Chosen over scope-test so the invariant holds for ALL nodes and the graph stays internally consistent rather than tolerating dead half-states.

Axis B — regenerate the 203 empty-description StandardNameSource nodes, or prune the stale ones first?

14 StandardName + 203 StandardNameSource carry literal '' descriptions and 12 SN have embedded_at set but no vector; the affected SNs skew to dead states (superseded/exhausted) and the sources include stale/failed/skipped/vocab_gap. An empty description on a dead source is noise, not a gap — triage first: prune/normalize dead sources, regenerate genuinely-live empty descriptions through the SN pipeline (sn run), never hand-write text. Investigate why '' (vs null) is being written and fix at source if it is a worker bug.
