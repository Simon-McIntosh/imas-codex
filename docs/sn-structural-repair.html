§0 — Premise: most "structural defects" are stale data, not live bugs

Investigating the sibling-drift complaint (sn-family-harmonization) surfaced a class of structural defects in the SN graph: names that don't parse, projection children parented under the wrong node, and parent/child physics-domain mismatches. Research established the key finding: the current derivation.py and parents.py code already produces the correct edges and parse columns — the bad edges/columns are leftovers from older pipeline versions. So this is largely a deterministic, ~$0 re-derivation + backfill job, not a redesign.

The motivating case: toroidal_mhd_mode_number (physics_domain=transport, column-null) is wrongly acting as the projection-parent of poloidal_mode_number and radial_mode_number (both magnetohydrodynamics). A toroidal-specific name cannot be the root of a poloidal projection. Running current derive_edges() on those children correctly points them at the bare base mode_number — the live graph just carries a stale edge migrated during an old refine cycle (persist_refined_name migrates source edges, graph_ops.py:8675).

§ Decisions

The correct bare-base parents (mode_number, *_diffusivity) are currently superseded. How should edge re-derivation handle that?

The bare base was superseded by a refine cycle whose successor turned out to be one of its own axis-projections — that is exactly the bug. Un-supersede is likely correct (the bare base is the legitimate vector parent and Clause B re-admits it once the projection edges point at it), but it must be checked against the REFINED_FROM chain so we don't resurrect a genuinely-bad name. Needs a dry-run inspection of each affected chain before choosing.

Run the repair as inline Cypher migrations (graph shell) or as a scoped CLI phase?

AGENTS.md says run migrations as inline Cypher, not scripts/migrate_*.py — favours inline for the one-off backfill/edge work. But edge re-derivation must call derive_edges + the admission gate (Python, not pure Cypher), so a small scoped CLI phase that reuses those functions may be cleaner and re-runnable. Likely: backfill + renames inline; edge re-derivation via a scoped CLI entry that calls the existing Python.
