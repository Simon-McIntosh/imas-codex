§0 — Premise: stale data AND a live non-idempotency bug

Reframing (2026-06-25, handoff f-snch-003 + live investigation). The original premise below ("stale data, not a redesign") is only half right. The stale edges are real, but the deeper finding is that the four startup functions (rederive_structural_edges → seed_parent_sources → normalize_derived_parent_lifecycle → structural_accept_derived_parents, loop.py:902) are not idempotent: every sn run startup mints fresh placeholder parents and orphans others even on a stable name set. Three bugs: (B1) accretion — _write_standard_name_edges MERGE-adds derived HAS_PARENT edges and never deletes a child's stale ones (this is the diffusivity-8 + mode_number tangles); (B2) admission ping-pong — Clause-B + the single-child-shadow veto read the accreted topology, so mint↔delete oscillates per startup (~154/rotation); (B3) unreaped zombies — normalize_derived_parent_lifecycle only re-checks parents that still have a child, so childless derived parents accumulate (live: 327, of which 283 carry orphaned docs). So §2b is a code fix + one-time data repair, not pure re-derivation.

Investigating the sibling-drift complaint (sn-family-harmonization) surfaced a class of structural defects in the SN graph: names that don't parse, projection children parented under the wrong node, and parent/child physics-domain mismatches. Research established the key finding: the current derivation.py and parents.py code already produces the correct edges and parse columns — the bad edges/columns are leftovers from older pipeline versions. So this is largely a deterministic, ~$0 re-derivation + backfill job, not a redesign.

The motivating case: toroidal_mhd_mode_number (physics_domain=transport, column-null) is wrongly acting as the projection-parent of poloidal_mode_number and radial_mode_number (both magnetohydrodynamics). A toroidal-specific name cannot be the root of a poloidal projection. Running current derive_edges() on those children correctly points them at the bare base mode_number — the live graph just carries a stale edge migrated during an old refine cycle (persist_refined_name migrates source edges, graph_ops.py:8675).
