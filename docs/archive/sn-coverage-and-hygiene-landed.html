§1 — Derived-parent enrichment (coverage)

The deadlock. A derived parent (origin='derived') is materialized with a placeholder description "(deterministic parent — description pending LLM enrichment)". That placeholder excludes it from REVIEW_NAME (the claim requires a real description for the semantic-similarity check), so it never earns a reviewer_score_name; and claim_generate_docs_batch gates on reviewer_score_name IS NOT NULL, so it never earns docs. The parent is parked at name_stage='accepted' (held), invisible to both axes — neither reviewable nor documentable.

The fix — a seventh sn run pool, ENRICH_PARENTS. An async pool (the user's choice: in-loop, claiming work from the graph, sharing the single BudgetManager like every other pool — no new CLI surface). For each claimed placeholder parent with ≥1 live child it: fetches the parent's children, synthesizes a real description generalized over those children (grounded strictly on them — no invented physics), embeds it locally (free), and routes name_stage→'drafted' so it flows review → accept → docs. Childless derived parents are legitimately unscoped and never claimed. Model: get_model("sn-parent-enrich") (compose-tier deepseek-v4-flash, ~$0.0017/parent; generate_docs rewrites the full long-form documentation downstream).

Backlog (live, 2026-06-23): 723 placeholder derived parents — 441 enrichable (have ≥1 live child), 282 childless (correctly skipped). The plan's earlier "612/50" split had drifted as the graph evolved; the design handles whatever the live split is (enrich those with children, skip childless).

✓ landed 2026-06-23

§2 — Bare-bracket normalize at source (hygiene)

§2a (accept-path, the must-have): bare [name] brackets are now normalized at the moment a doc promotes to accepted in persist_reviewed_docs — scoped to the single node (no full-catalogue scan), linking the token if it is a live SN else stripping it. No accepted doc can carry a broken bracket regardless of when it was written, eliminating the per-cycle manual resolve_doc_links sweep. §2b: the post-drain resolve_doc_links reconcile already runs as the true final step in run_sn_pools — retained as a belt-and-suspenders net for cross-references to names accepted later. §2c: regression tests (test_bare_doc_link_normalize.py) assert a doc accepted with a bare [live] is linked and [dead] is stripped, and that the normalize fires only on the accepted transition.

Proven on live content: a generate_docs output for hydrogenic_ion_mass_density carried a properly-formed [ion_mass_density](name:ion_mass_density) cross-reference with zero bare brackets.

What's next

Full drain (operator campaign, the user's stated target). The mechanism is shipped and verified; draining all 441 enrichable parents through enrich → name-review → accept → docs → docs-review → accepted → export is a multi-rotation sn run --flush run.

⚠ Budget caveat (pre-existing soft-cap behavior, surfaced during verification): the --cost-limit stops admission but lets in-flight batches finish during the grace window. With 64 review_docs replicas this overshot materially — a $1 names cap spent $1.70 (1.7×) and a $2 docs cap spent $5.37 (2.7×). Budget the full drain with ~2–3× headroom, or lower [sn-pools].review-docs-replicas for tighter control. This is not introduced by this work (docs-only does not even run enrich) — it is the existing budget-watchdog + grace-window characteristic.

Unblocks: sn-expert-hints §6 dog-food (edit a real-description parent), sn-structural-repair, and sn-family-harmonization. The 282 childless placeholder parents are candidates for sn-structural-repair's stale-edge cleanup.
