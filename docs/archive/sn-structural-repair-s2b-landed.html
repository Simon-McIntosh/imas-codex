§2b — derived-parent idempotency fix + structural data repair (landed 2026-06-25)

The handoff f-snch-003 reframed §2b: the derived-parent churn was a live non-idempotency bug, not stale data. Root-caused, fixed in code, then repaired the live graph to a fixpoint.

The fix — four changes in graph_ops.py

  1. Reconcile, not accrete — _write_standard_name_edges deletes each child's stale derived HAS_PARENT edges and writes exactly the current admitted derivation.
  2. Live names only — rederive_structural_edges derives structure from non-superseded/-exhausted names and clears derived edges originating FROM dead names (they were re-minting reaped parents).
  3. Zero-incoming reaper — normalize_derived_parent_lifecycle reaps truly-orphaned origin='derived' parents (reaping on "no live child" ping-ponged: dead children re-derive the parent each startup).
  4. Batch-authoritative admission (decisive) — _filter_admissible_parents(..., full_rebuild=True) evaluates Clause B + the single-child shadow veto from the batch alone, not transient pre-reconcile graph topology. That graph dependence flipped the shadow-veto verdict for qualifier single-child parents (absorbed_, total_, average_…) → a clean 2-cycle. Batch-authoritative ⇒ admission is a pure function of the live-name set ⇒ convergence.

Also: _rewire_has_parent_off_superseded no longer migrates children onto a REFINED_FROM successor that is a projection-of-self of the base. Caught a Neo4j-2026 NOT IN syntax bug that passed the string-matching unit stub but would have failed live.
