The unit gate (drop-and-triage)

A provenance pair is dropped only on a known disagreement: the DD path carries a HAS_UNIT edge whose unit canonically conflicts with the name's, judged by units_agree over the ISN canonical_unit authority plus the DD-unit-bug exception record. A DD path with no unit edge is a DD-completeness gap (nothing to disagree with) and is attached — the same predicate the soundness invariant (test_dd_attachment_unit_agrees_with_name) already asserts, so edge and invariant stay self-consistent. 112 pairs were dropped and logged for the unit-curation triage — genuine conflicts such as ion/state/z_min tagged e (elementary charge) against a dimensionless charge number, and equilibrium/…/gm4/gm5 metric coefficients. They are surfaced, never forced into the index.

Discovery residual (measure-first)

1 530 SN-eligible DD nodes (of 10 288) carry no producing source and no edge. All 1 530 have no StandardNameSource at all — they are ordinary extraction backlog, not a distinct discovery mechanism. The plan's "→ all z_n" concern is served by the backfill already (every sibling path is its own source): ion_atomic_number reaches all 85. So the discovery-axis decision locked to extract-sources: any genuine same-quantity gap closes when the ordinary extraction pipeline gives the node a source, after which the now-always-on reconcile materializes its edge automatically. No orphan HAS_STANDARD_NAME edge is ever authored without a source behind it, and no dedicated discovery predicate was built.

Not a catalog change

The published imas-standard-names-catalog SPA reads the exported sources field, built from PRODUCED_NAME (already complete) — it never reads HAS_STANDARD_NAME. This repair is purely codex-internal: it restores the 1-hop index that review-context enrichment, source eligibility, coverage, dedup, derived-parent admission, and the MCP edge-walkers all depend on.
