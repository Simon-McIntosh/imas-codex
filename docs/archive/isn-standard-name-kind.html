ISN StandardNameKind Assessment

This archive preserves a boundary-setting decision rather than an implementation recipe. The markdown source concluded that StandardNameKind should stay as scalar, vector, and metadata; the new geometric-versus-physical distinction belongs in DD node classification, not in ISN’s grammar-level enum.

§1 — Main conclusion

Keep StandardNameKind unchanged. The source plan argued that the enum answers a different question from DD node categories: mathematical form and grammar validation, not pipeline participation or physical provenance. That conclusion remains the key historical outcome.

  • Why: the grammar already distinguishes structured geometric forms through its own branches, while many geometric quantities still use the open physical-base path.
  • What not to do: do not denormalize DD geometry labels into a singular StandardName field unless a future search path proves the join too expensive.
  • Optional escape hatch kept in the archive: if denormalization is ever required, it should be a recomputed multivalued source-category set, not a singular unstable flag.

§2 — Cross-reference preserved

The file directly informed DD Unified Classification & Enrichment Pipeline by clarifying that codex owns DD category semantics and ISN owns grammar and validation. The archived markdown also kept an important warning: compose prompts need category-aware hints so geometric quantities choose the correct grammar branch.

§3 — What was later superseded

The source itself recorded that only part of its assessment remained fully current. The recommendation to keep StandardNameKind unchanged remained valid, while its old “vector kind is effectively unused” conclusion was superseded by later vector-hierarchy work in the Standard Names planning stream. This archive intentionally keeps both facts visible.
