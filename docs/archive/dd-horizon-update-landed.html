What shipped (code)

imas dd update — the horizon roll-forward command

Auto-detects the graph's is_current horizon and the latest installed DD version, refuses to lower the horizon, bumps [tool.imas-codex.data-dictionary].version in pyproject, then runs imas dd build in a fresh subprocess — the module-global imas_codex.dd_version is resolved at import time, so the bumped horizon only re-keys is_current + the reconciler in a clean interpreter. It captures the SN provenance-link counts before and after and fails the command if any count decreased or any link dangles. Optional --to-version pins a target.

Footgun removal — the three IMASNode-deleting paths

Deleted precisely: the extracted target of --reset-to (dropped from the flag's choices; _reset_to_extracted's DETACH DELETE removed), --force (which implied extracted; the force field + skip_build_hash property gone from DDBuildState), and the imas dd clear command + clear_dd_graph. The four non-destructive --reset-to built|enriched|refined|embedded targets stay verbatim (project convention; property-clear + re-run on existing nodes, never delete) — needed when a prompt/model change is invisible to the content hash. No code path deletes an IMASNode now short of dropping the whole graph instance. A from-scratch build targets a fresh empty instance (graph init); a build-logic change re-MERGEs via the _BUILD_SCHEMA_VERSION bump.

SN-link guardrail

sn_link_guardrail.py: capture_sn_link_counts() snapshots the three provenance-edge counts + the dangling count (a source naming a dd_path with no FROM_DD_PATH → IMASNode); check_sn_links_safe() returns violations (empty == safe: non-decreasing edges, dangling == 0). The -m graph test asserts the pure comparison logic plus the live dangling==0 invariant. The reattach decision (D-reattach = stay-pinned) collapses, with the footguns gone, into this dangling==0 check — no code path can strand a link, so the "repair" is a check that must always pass.

The roll (operational)

Config was already at 4.1.1 and the graph's is_current was 4.1.1 — but 4.1.1 was extracted-but-unfinished (no build_hash, no clusters). The roll therefore finished 4.1.1 rather than adding a new version. Result:

  • 4.1.1 now fully built: build_hash written, 2950 clusters (23324 IN_CLUSTER edges), embeddings 9480→9503.
  • Lifecycle stamped: the reconciler marked historical paths removed (the null-lifecycle backlog auto-resolved, as the plan predicted) — never deleting.
  • Hash no-op confirmed: of 61k nodes only 34 were enriched (LLM) and 23 re-embedded — the unchanged tens of thousands cost nothing.
  • Incremental spend $0.81 (34 enrich + 23 embed + 157 cluster labels) vs ~$31.22 cumulative full-enrichment baseline — ~2.6%.
  • Elapsed 655 s (~11 min), dominated by the full cluster recompute + labeling (accepted by design).
  • SN links: before = after — 8763 FROM_DD_PATH, 2454 HAS_STANDARD_NAME, 2316 VOCAB_GAP, 0 dangling. Guardrail passed automatically: "Horizon rolled to 4.1.1. SN provenance links intact."
