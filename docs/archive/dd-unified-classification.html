DD Unified Classification & Enrichment Pipeline

This archive keeps the broad design that unified several DD threads: category repair, quantity-versus-geometry semantics, search fallout from stale 'data' filters, enrichment-model choice, and the bridge to multi-pass refinement.

§1 — Durable taxonomy decisions

  • quantity was split from geometry. Machine and diagnostic hardware geometry became a first-class searchable and embeddable category instead of being lumped into a monolithic quantity bucket.
  • Geometry detection became structural. The plan replaced fragile leaf-name heuristics with path-pattern plus spatial-unit rules, while explicitly excluding plasma outputs such as boundary and separatrix data.
  • Coordinate handling stayed separate. Coordinates remained enrichable and searchable but not embeddable or SN-extractable.
  • The assessment of StandardNameKind stayed orthogonal. The plan preserved the boundary that DD category decisions belong in codex, while ISN grammar and StandardNameKind remain separate concerns; see ISN StandardNameKind Assessment.

§2 — Execution work the markdown preserved

  • Dirty-worktree recovery was part of the plan. The source explicitly documented how reverted search changes had broken DD search by reintroducing hardcoded 'data' filters.
  • Classifier repair was specific. The archive keeps the path-aware fix for coordinate targets, the HAS_PARENT relationship correction, and the requirement to update tests before graph repair.
  • MCP/search augmentation was bundled in. Category-aware filters, coordinate metadata display, and geometry-vs-physical shorthands were all treated as part of the same classification cleanup.
  • Model choice was re-evaluated empirically. The markdown preserved the finding that current DD enrichment had been done with flash-lite and that moving to sonnet was a dramatic quality upgrade for only marginal extra cost.

§3 — Relationship to later refinement work

The source explicitly introduced DD Multi-Pass Enrichment as the follow-on design: first fix the category system and model threading, then add a second refinement pass with a new refined status. That dependency is preserved here because the refinement plan assumes the unified classifier, enrichable-category sets, and CLI model wiring from this plan.
