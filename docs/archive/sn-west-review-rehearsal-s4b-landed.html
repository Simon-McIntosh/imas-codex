§4b — The fold-back receipt tag (landed 2026-07-22)

Two events complete a release: the human merges the catalog PR (durably recorded by GitHub) and the maintainer folds it back into the graph (sn merge). The second was recorded nowhere durable, so a merged-but-not-folded release was indistinguishable from a folded one. The receipt closes that gap. D-final-shape (locked) named the version tag as the fold-back receipt, owned by sn merge; §4b later added a grounded human summary below the contract block. The contract

After a successful non-dry fold-back, sn merge creates an annotated tag on the merge commit (batch: vX.Y.ZrcN; final: vX.Y.Z, derived from the PR head branch review/<rc> or release/<version>) and pushes it to the PR's target repo remote (fork for a batch RC, upstream for a final — matched from the PR URL's owner/repo against the checkout's remotes). Tag-on-merge-commit also resolves the ordering problem a release-time tag would have had — a tag cut before the merge points at a commit the target repo never receives.

The tag message layout — contract first, prose below a separator:

graph-merged: 2026-07-22T09:14:03+00:00
pr: #7 https://github.com/<org>/<catalog>/pull/7
batch: v0.2.0rc65.sn_names.yaml
outcomes: approved=1 auto_approved=310 contested=1

---

<grounded human summary of what review changed>
  • Machine-readable, deterministic prefix. The idempotency check parses only the first line (graph-merged:). A tag carrying that marker ⇒ catalog and graph are in sync for the version, so sn merge refuses a second fold-back (clear message, nonzero exit).
  • Inverse. sn merge --undo deletes the tag (local + remote) — its absence again means "merged but not folded back".
  • Push failures are reported, never silent. A failed push rolls back the local tag so no local receipt exists without a remote counterpart.

The grounded summary

Below the separator, build_merge_notes (the sn-release-notes seat — no new seat) synthesizes a factual account of what review did to the batch, from four evidence sources recovered from the PR itself via gh:

  1. the PR description;
  2. the full PR conversation — review comments and threads (gh pr view --json body,comments,reviews);
  3. every commit message that went into the PR (--json commits);
  4. the review-delta diff — the diff between the PR's original content (the first commit of the PR) and the merged state, scoped to standard_names/, i.e. exactly what the reviewers changed.

Grounded summarisation only — never invention. The summary is prose below the separator; it is never parsed and never blocks: on any synthesis failure the tag is written with the deterministic block alone, and --no-notes skips it entirely. Downstream, the catalog site/CI deploys versions from tags, so a deployed catalog version now implies graph consistency.
