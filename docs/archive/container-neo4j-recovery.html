Fix: Neo4j Database Unavailable in Container

This archive keeps the root-cause analysis for the container incident where Azure returned HTTP 200 from /health while all graph statistics were null. The preserved conclusion is that the DBMS came up before the neo4j database finished recovery, so the MCP server started against an offline database.

§1 — Root cause and evidence

  • Readiness bug: the entrypoint checked only HTTP readiness, not Bolt queryability.
  • Environment asymmetry: CI runners recovered the 1.9 GB graph quickly on local SSD, while Azure’s slower storage and tighter memory budget made recovery take minutes or stall permanently.
  • Non-causes preserved for future debugging: auth was not the issue, and the graph dump was already baked into the image correctly.

§2 — Preserved recovery design

  1. Build-time recovery: pre-start Neo4j in the graph-loader stage so WAL replay and system-database creation happen on the fast build host, not on slow Azure storage.
  2. Runtime Bolt gate: wait first for HTTP/DBMS readiness, then for a successful cypher-shell "RETURN 1" probe before starting the MCP server.
  3. Optional hardening: a modest heap increase and an explicit graph-status field in the health response.

The important preserved operational rule is that container startup must verify database readiness, not just process liveness.

§3 — Validation target kept from the markdown

The archived acceptance bar was straightforward: build succeeds, container startup exposes non-null graph stats within seconds, runtime logs no longer show fresh recovery work, and Azure health checks report populated graph metrics. Keep this file as the incident RCA and the rationale for any future entrypoint or image-startup changes.
