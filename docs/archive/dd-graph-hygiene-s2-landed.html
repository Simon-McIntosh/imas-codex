Root cause (live-verified, reframed the plan's snapshot)

The plan's authoring snapshot assumed the ~2457 embedding_text-but-no-vector nodes were legacy classified nodes that could be reset to refined so the embed worker reclaims them. The live graph disproved that premise:

  • All 2824 text-no-vector nodes are in non-embeddable categories (representation 2045, fit_artifact 330, structural 276, coordinate 98, metadata 75) — zero are embeddable. Embedding is deliberately restricted to EMBEDDABLE_CATEGORIES (quantity/geometry/identifier) to keep semantic-search noise low, so these nodes correctly have no vector. Resetting them to refined is a no-op — the embed worker filters on embeddable categories and would never claim them.
  • No current code path persists embedding_text to a non-embeddable node (worker-embed claims embeddable categories; update_path_embeddings runs under a clusterable filter / embedding IS NOT NULL). The 2824 are legacy cruft from an earlier category scheme. Generation is already gated — so the fix is to clean the graph, exactly as expected.
  • The 198 vector-but-no-text nodes are all identifier (embeddable, correctly embedded) hit by a real persistence bug: embed_identifier_nodes computed the source text but never wrote it.
