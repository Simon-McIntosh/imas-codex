Search Recall, Scoring, and Output Enrichment

This archive preserves the search-improvement program that explained why the graph-backed DD search underperformed the older file-backed server on recall while already winning on some precision-oriented semantic queries. The key diagnosis was cumulative candidate loss across approximate vector search, aggressive filtering, and weak score mixing.

§1 — Foundation fixes worth keeping

  • Remove the BM25 score floor. The markdown identified the 0.7 floor as a distortion that compressed text relevance and let weak text matches outrank better vector hits.
  • Switch hybrid fusion to RRF. Reciprocal Rank Fusion replaced ad-hoc additive mixing and became the preserved ranking recommendation.
  • Over-fetch more vector candidates. The plan argued for raising the vector candidate limit so broad physics queries do not lose most relevant paths before scoring starts.
  • Relax the documentation filter. Once enriched descriptions existed, search should not exclude otherwise good nodes simply because raw DD documentation was short.

§2 — Graph-native capabilities the plan documented

  • Lucene advanced queries: field boosting, fuzzy matching, phrase proximity, and prefix queries.
  • Cluster and hierarchy signals: cluster-consensus boosts, sibling/parent proximity, and coordinate-based cross-IDS discovery.
  • Agent-facing output enrichment: children previews, dual description/raw-DD display, provenance tags for search channels, and breadcrumb-style parent context.

The source plan also preserved a useful architectural note: some items later moved into a successor search-cluster/output evaluation effort, but the rationale for graph-native retrieval and richer result formatting still belongs in the historical record.

§3 — Evaluation discipline preserved

The markdown did not stop at ranking ideas; it specified a permanent evaluation loop. The archived record keeps the call for a larger ground-truth corpus, a parameterized harness, DoE-based tuning of hybrid weights and limits, and CI regression gates for future graph_search.py changes. It also keeps the verified-capabilities checklist showing that Lucene syntax, reverse HAS_PARENT traversal, cluster lookups, and coordinate-sharing queries were all already proven on the live Neo4j graph.
