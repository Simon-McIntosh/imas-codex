The seeding defect (fixed, no spend)

The --focus/--batch seeders build StandardNameSource dicts and hand them to merge_standard_name_sources, which pins each DD source to an immutable snapshot of its DD semantics. Two failures surfaced in sequence:

  1. New source, no version. A genuinely-new source (e.g. interferometer/electrons_n, added to the corpus by the dimensionless-signal classifier fix) reached the pin guard with no dd_version; the guard refuses to infer latest, so the whole mop-up aborted before any spend.
  2. Re-seed, wrong version. A first patch that stamped the current version on every source then tripped the opposite guard: an already-pinned source (pinned to 4.1.0, older than the current 4.1.1) looked like a re-pin request and raised.

The fix threads a batch-level default_dd_version through merge_standard_name_sources to _pin_dd_source_snapshots: it pins only genuinely-new sources, while re-seeds keep their immutable stored pin. Per-source dicts carry no version (the documented re-seed contract), and the per-source mismatch guard is preserved. Applied to all four seeders (the CLI focus/batch path, both pool_adapter seeders, and the extraction worker), so the fragility is closed everywhere, not just on the path this rehearsal exercised. Regression + guard tests (unit + graph) cover new-source pin, re-seed default-ignore, and the version-free dict shape; full SN suite green.

Mechanically the mop-up now runs cleanly: it seeded the 150 gap sources (355 − 205 already-accepted) and launched all pools before hitting the billing wall.

The blocker: OpenRouter key total-limit

OpenRouterException: {"error":{"message":"Key limit exceeded (total limit)."}}  → HTTP 403

Every compose / docs / review call returns 403, so the run recorded stop_reason=no_eligible_work, cost_spent=$0.0, names_composed=0. The 150 gap sources were reset to extracted by seeding and then marked failed when their calls 403'd — a recoverable state (they carry no accepted name, so a gap-only re-run retries them). The proxy itself is healthy; the limit is at the OpenRouter account, which llm status cannot see because it makes no API calls.
