COCOS Diagnostic Tools — EFIT Input Spec

Date: 2025-01-28 From: EFIT (IO) team To: imas-codex team Status: Request for comment

1. EFIT Context

1.1 The Problem We Hit

In April 2026, EFIT shipped a sign-convention bug (cocosIpSign returning −1 instead of +1 for DDv3 input) that persisted for ~50 days across 4 corrective commits. The bug was independently encoded in both C++ (src/imas.cpp) and Python (efit/cli/cocos_canonical.py), showing it was a conceptual error, not a typo.

Root cause: the developer assumed σ_Ip differs between COCOS-11 (DDv3) and COCOS-17 (DDv4). It does not — σ_Ip is tied to σ_Rφz (coordinate handedness), not σ_Bp (poloidal field sign), and σ_Rφz = +1 in both conventions.

1.2 Why We Need a Shared Tool

EFIT cannot be the only IMAS consumer that makes sign-convention mistakes. The Sauter COCOS framework has 4 parameters, 16 valid index values, and non-obvious derived sign relationships. Any code that reads multi-DD-version data must implement COCOS transforms, and every implementation is a new opportunity for the same class of bug.

A shared, authoritative COCOS detection tool in imas-codex would: 1. Let any consumer validate their data before processing 2. Detect mixed-convention datasets (e.g. magnetics in COCOS-11 + wall in COCOS-17) 3. Serve as the reference implementation of the Sauter framework 4. Power CI-time data-quality checks across the IMAS ecosystem

1.3 What EFIT Needs

EFIT will be the first consumer. Our requirements:

Capability Priority Rationale
Detect σ_Bp from data signs P0 Distinguishes COCOS-11 from COCOS-17; our actual bug
Validate ids_properties.cocos consistency P0 DDv4 datasets declare COCOS; check data matches declaration
Per-IDS sign analysis P1 Mixed-convention datasets are real (DDv3 machine description + DDv4 measurements)
Report non-conforming paths P1 Narrows diagnosis to “flux_loop is flipped” rather than “something is wrong”
Detect σ_Rφz P2 Required for COCOS-2/4/6/8 detection; EFIT currently only handles 11/17
exp_Bp detection P3 Requires absolute scale reference; defer unless low-cost

2. Proposed API

2.1 Core Data Model — CocosReport

@dataclass
class CocosObservation:
    """Single sign observation from an IMAS data field."""
    path: str                      # Full DD path
    ids_name: str                  # IDS containing this path
    transformation_type: str       # DD cocos_transformation_type
    observed_sign: int             # -1, 0 (indeterminate), or +1
    time_index: int | None = None  # If time-dependent
    note: str = ""                 # Human-readable context

@dataclass
class CocosCandidate:
    """Scored COCOS candidate from the detector."""
    cocos_index: int               # 1–18
    sigma_bp: int                  # +1 or -1
    sigma_rphiz: int               # +1 or -1
    sigma_rhotp: int               # +1 or -1
    exp_bp: int | None             # 0, 1, or None (indeterminate)
    hamming_distance: int          # Number of disagreeing observations
    confidence: float              # 1.0 - (hamming_distance / n_observations)

@dataclass
class CocosReport:
    """Complete COCOS detection report for a dataset."""
    uri: str                       # IMAS URI of the analysed dataset
    dd_version: str                # DD version of the dataset
    declared_cocos: int | None     # From ids_properties.cocos (DDv4 only)
    detected_cocos: CocosCandidate | None  # Best-fit detection result
    alternatives: list[CocosCandidate]     # All candidates sorted by score
    observations: list[CocosObservation]   # Raw sign observations
    non_conforming: list[CocosObservation] # Observations disagreeing with best fit
    matrix_rank: int               # Rank of transformation matrix (max 3)
    n_observations: int            # Total non-indeterminate observations
    warnings: list[str]            # Human-readable caveats
    is_consistent: bool            # True if all observations agree with one COCOS
    timestamp: str                 # ISO 8601

2.2 MCP Tool Spec — detect_cocos

Tool: detect_cocos

Description:
  Detect the COCOS sign convention of an IMAS dataset by inspecting
  the signs of COCOS-sensitive fields. Returns a structured report
  with the best-fit COCOS index, confidence score, and per-path
  diagnostics.

Args:
  uri (str, required):
    IMAS URI pointing to the dataset.
    Example: "imas:hdf5?path=/data/WEST/58072/"

  dd_version (str, optional):
    DD version to use for path metadata lookup.
    Default: read from ids_properties.version_put.data_dictionary

  ids_list (list[str], optional):
    Restrict analysis to specific IDSs.
    Default: all IDSs present in the dataset.
    Example: ["magnetics", "pf_active", "equilibrium"]

  time_index (int, optional):
    Analyse a specific time slice only.
    Default: first non-vacuum slice (auto-detected).

  include_observations (bool, optional):
    Include per-path sign observations in output.
    Default: true

Returns:
  CocosReport as JSON.

2.3 Python Library API

from imas_codex.cocos import detect_cocos, CocosReport

# Basic usage
report: CocosReport = detect_cocos("imas:hdf5?path=tests/data/imas/WEST/58072/")

# Restricted to specific IDSs
report = detect_cocos(uri, ids_list=["magnetics", "pf_active"])

# Check consistency
if not report.is_consistent:
    for obs in report.non_conforming:
        print(f"  Non-conforming: {obs.path} ({obs.transformation_type})")

3. Required DD Metadata

The detector needs the following metadata per COCOS-sensitive DD path:

Metadata Field Source Purpose
cocos_transformation_type DD node property Maps path to sign-bit dependency
data_type DD node property Determines how to extract sign (FLT_1D → array vote)
lifecycle_status DD node property Skip obsolescent/alpha paths
coordinate1..N DD node property Identifies time dimension for slicing

3.1 Transformation Type → Sign Matrix

The detector maps each cocos_transformation_type to a row in the 3-column sign matrix (σ_Bp, σ_Rφz, σ_ρθφ):

Type σ_Bp σ_Rφz σ_ρθφ Net DDv3→DDv4 factor
psi_like 1 0 0 −1
dodpsi_like 1 0 0 −1
ip_like 0 1 0 +1
b0_like 0 1 0 +1
q_like 0 0 0 +1 (composite, net invariant for DDv3→DDv4)
one_like 0 0 0 +1
tor_angle_like 0 0 0 +1
pol_angle_like 0 0 1 +1

3.2 MCP Requirements

The detector needs to look up DD metadata at runtime. The imas-codex DD MCP tools already provide everything needed:

Need MCP Tool Call Example
Path→transformation_type search_dd_paths query="flux_loop flux", ids_filter="magnetics", cocos_transformation_type="psi_like"
All COCOS-tagged paths get_dd_cocos_fields ids_filter="magnetics"
Path existence check check_dd_paths paths="magnetics/flux_loop/flux"
Version-specific lookup All tools support dd_version= Filter by 3 or 4

For a compiled, fast-path implementation: bundle a static lookup table derived from get_dd_cocos_fields(ids_filter=None) at package build time.

4.2 Phase 2 — MCP Tool

Register detect_cocos as an MCP tool in the imas-codex server. The tool wraps the Python library and returns CocosReport as JSON.

4.3 Phase 3 — EFIT Integration

EFIT calls detect_cocos() as a pre-flight check before reading IMAS input:

# In efit CLI pre-flight:
from imas_codex.cocos import detect_cocos

report = detect_cocos(uri, ids_list=["magnetics", "pf_active", "tf"])
if not report.is_consistent:
    raise ValueError(f"COCOS inconsistency detected: {report.non_conforming}")
if report.detected_cocos.cocos_index not in (11, 17):
    raise ValueError(f"Unsupported COCOS: {report.detected_cocos.cocos_index}")

5. Corner Cases

5.1 Vacuum / Zero-Ip Slices

When |Ip| < threshold (vacuum or field null), most sign observables are indeterminate. The detector should: - Skip to the first slice with |Ip| > threshold - If all slices are vacuum: return detected_cocos = None with a warning

5.2 Mixed-Convention Datasets

Real datasets may contain IDSs written at different COCOS conventions: - Machine-description IDSs (wall, pf_active geometry) from one source - Measurement IDSs (magnetics, mse) from another

The detector should support per-IDS analysis and report the convention for each IDS independently.

5.3 DDv4 ids_properties.cocos Validation

When ids_properties.cocos is present: 1. Use it as the declared convention 2. Run sign detection anyway 3. Report any disagreement between declared and detected as a data quality warning, not a hard error

5.4 Sparse Data

Some IDSs may carry very few COCOS-sensitive fields. The detector should report matrix_rank so the caller can assess detection reliability: - Rank 0: no COCOS information (e.g. wall-only dataset) - Rank 1: one sign bit determined (e.g. tf-only gives σ_Rφz) - Rank 2: full detection for COCOS-11/17 (σ_Bp + σ_Rφz determined)

5.5 Array Voting for Multi-Element Fields

Fields like flux_loop[i]/flux/data contain N values with independent signs. The detector uses majority voting: - Count positive and negative non-zero values - If unanimous: strong observable - If mixed but majority > 2/3: weak observable (flag in report) - If tied or near-tied: indeterminate (s = 0)

6. EFIT Interface Plan

6.1 EFIT CLI Command

EFIT will wrap detect_cocos() as efit cocos-detect:

uv run efit cocos-detect "imas:hdf5?path=tests/data/imas/WEST/58072/"

This is an EFIT-side deliverable, not an imas-codex deliverable.

6.2 CI Integration

EFIT will add a CTest fixture that runs COCOS detection on every test dataset before the solver:

add_test(NAME cocos-check/${TEST_NAME}
  COMMAND ${CMAKE_COMMAND} -E env ${EFIT_UV_ENV}
    uv run efit cocos-detect "${URI}" --assert-consistent)
set_tests_properties(cocos-check/${TEST_NAME} PROPERTIES
  LABELS "IMAS;COCOS"
  FIXTURES_SETUP cocos_check_${DATASET})

6.3 Solver Pre-Flight (Future)

Long-term, the C++ solver could call the detector before reading IMAS input and auto-set its internal COCOS parameters from the detection result rather than from the DD version string. This eliminates the entire class of “assumed σ_X from DD version” bugs.

7.1 Acceptance Criteria for imas-codex Deliverables

  1. detect_cocos("imas:hdf5?path=<WEST_58072_DDv3>") returns COCOS-11
  2. detect_cocos("imas:hdf5?path=<WEST_58072_DDv4>") returns COCOS-17
  3. Injecting a sign flip on magnetics/ip produces is_consistent = False with the flipped path in non_conforming
  4. Empty / all-vacuum dataset returns detected_cocos = None
  5. DDv4 dataset with ids_properties.cocos = 17 and consistent data returns declared_cocos = 17, detected_cocos = COCOS-17, is_consistent = True
  6. DDv4 dataset with ids_properties.cocos = 17 but psi_like signs matching COCOS-11 returns is_consistent = False with a diagnostic warning
